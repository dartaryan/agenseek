## 🎯 למי מדריך זה מיועד

**מעצבי משחקים (Game Designers):**
- מעצבים שיוצרים מכניקות משחק, מערכות ותיעוד עיצוב
- אחראים על חוויית השחקן, איזון משחק ו-Core Loops
- צריכים ליצור GDD מקיף, תיאור מכניקות ועיצוב נרטיבי

**מפתחי משחקים (Game Developers):**
- מתכנתים שמממשים מכניקות משחק במנועים כמו Unity, Unreal, Godot
- אחראים על קוד Gameplay, אופטימיזציה וביצועים
- צריכים לממש Stories, לבצע Code Reviews ולבדוק איכות קוד

**ארכיטקטי משחקים (Game Architects):**
- ארכיטקטים המתכננים מערכות טכניות גדולות למשחקים
- אחראים על ארכיטקטורת Multiplayer, מנועים ו-Scalability
- צריכים לקבל החלטות טכנולוגיות ולתכנן תשתיות

**מה תשיגו עם BMAD:**
- **תיעוד עיצוב מקצועי** - GDD, Narrative Design ו-Game Briefs ברמה גבוהה תוך דקות
- **ארכיטקטורה מבוססת ניסיון** - תכנון טכני מקצועי המבוסס על 20+ שנות ניסיון
- **פיתוח מובנה ומאורגן** - מעקב אחר Stories, Sprint Planning ו-Code Reviews
- **חיסכון בזמן אדיר** - מה שלוקח ימים של כתיבה ותכנון, BMAD מבצע בשעות

## 🚀 מסלול למידה מהיר

### שלב 1: היכרות ראשונית (יום 1)

**מטרה:** להבין מהו BMAD ואיך הוא משרת את פיתוח המשחקים שלך.

BMAD-METHOD הוא מערכת AI מבוססת אגנטים מיוחדים שמדמים צוות פיתוח משחקים מלא. במקום לכתוב לבד מסמכי GDD, לתכנן ארכיטקטורה או לנהל Stories - יש לך צוות וירטואלי של מומחים שמכירים את כל הפרוטוקולים והשיטות המובילות בתעשיית המשחקים.

**שלושת אגנטי המשחקים המרכזיים:**

1. **Samus Shepard (Game Designer)** - מעצב משחקים מנוסה שיוצר GDD, מכניקות ועיצוב נרטיבי
2. **Link Freeman (Game Developer)** - מפתח בכיר שמממש Stories, מבצע Code Reviews ומאופטם ביצועים
3. **Cloud Dragonborn (Game Architect)** - ארכיטקט ראשי שמתכנן מערכות, Multiplayer ותשתיות

**איך זה עובד?** כל אגנט מגיע עם תפריט וורקפלואים (workflows) - תהליכי עבודה מובנים שמובילים אותך צעד אחר צעד. לדוגמה:
- רוצים לעצב משחק חדש? → `@game-designer hi` → הריצו `*brainstorm-game`
- רוצים ליצור GDD? → `@game-designer hi` → הריצו `*gdd`
- רוצים לתכנן ארכיטקטורת Multiplayer? → `@game-architect hi` → הריצו `*create-architecture`

**תוצאה:** תבינו את הקונסטרוקציה הבסיסית - אגנטים + וורקפלואים = תהליכי עבודה מובנים לפיתוח משחקים.

### שלב 2: התקנה והגדרה (יום 1-2)

**מטרה:** להתקין את BMAD ולהגדיר את הסביבה לעבודה.

**התקנה מהירה:**
עיינו במדריך ההתקנה המפורט: [04-התקנה-והגדרה-מתקדמת.md](../04-התקנה-והגדרה-מתקדמת.md)

**צעדים עיקריים:**
1. התקנת המודול BMM (BMAD Method Module) - מכיל את כל אגנטי פיתוח המשחקים
2. הגדרת קובץ config.yaml - שם משתמש, שפת תקשורת ותיקיית פלט
3. טעינת BMad Master לראשונה - `@bmad-master hi`
4. בדיקה שהכל עובד - הצגת התפריט והרצת `*list-agents`

**טיפ חשוב:** BMAD עובד עם IDEs מבוססות AI כמו Cursor, Windsurf, Cline ו-Claude Code. וודאו שהסביבה שלכם תומכת בטעינת קבצי agents בפורמט Markdown.

**תוצאה:** BMAD מותקן ופועל, אתם מוכנים לטעון אגנטים ולהתחיל עבודה.

### שלב 3: אגנטים ווורקפלואים (שבוע 1)

**מטרה:** להכיר לעומק את שלושת אגנטי המשחקים המרכזיים ואת הוורקפלואים החשובים ביותר.

**יום 1-2: Samus Shepard - Game Designer**

Samus הוא נקודת הפתיחה לכל פרויקט משחק. הוא עוזר לכם לעבור מרעיון מעורפל ל-GDD ברור ומפורט.

**נסו את הוורקפלואים הבאים:**
- `*workflow-status` - לבדיקת מצב הפרויקט ומה הצעד הבא
- `*brainstorm-game` - Brainstorming מובנה לרעיונות משחק
- `*gdd` - יצירת Game Design Document מקיף
- `*narrative` - עיצוב נרטיב למשחקים מונחי סיפור

**תרגיל מעשי:** פתחו פרויקט משחק קטן (רעיון פשוט שאתם רוצים לממש). הריצו `*brainstorm-game` ולאחר מכן `*gdd`. תראו איך Samus מוביל אתכם דרך תהליך מובנה לתיעוד מקצועי.

**יום 3-4: Link Freeman - Game Developer**

Link לוקח את התיעוד שSamus יצר ומממש אותו בקוד. הוא מתמחה בפיתוח Stories ו-Code Reviews.

**נסו את הוורקפלואים הבאים:**
- `*workflow-status` - בדיקת Stories הממתינות לפיתוח
- `*develop-story` - מימוש Story מלא (קוד + tests)
- `*code-review` - Code Review יסודי לפני Merge
- `*story-done` - סימון Story כ-Done

**תרגיל מעשי:** קחו את ה-GDD שיצרתם עם Samus. השתמשו ב-Link כדי לממש Story קטן - למשל "מימוש מערכת תנועה בסיסית". תראו איך הוא מפרק את המשימה לקוד ובדיקות.

**יום 5-7: Cloud Dragonborn - Game Architect**

Cloud אחראי על התכנון הטכני הגדול - ארכיטקטורה, טכנולוגיות ומערכות מורכבות.

**נסו את הוורקפלואים הבאים:**
- `*workflow-status` - בדיקת מצב התכנון הטכני
- `*create-architecture` - תכנון ארכיטקטורה טכנית
- `*solutioning-gate-check` - אישור מעבר מתכנון למימוש

**תרגיל מעשי:** לקחו את אותו פרויקט קטן. העלו לCloud שאלה טכנית - למשל "האם להשתמש ב-Unity או Godot?" או "איך לתכנן Multiplayer ל-10 שחקנים?". תקבלו ניתוח מעמיק עם Pros/Cons.

**תוצאה:** תכירו לעומק את שלושת האגנטים, תבינו מתי להשתמש בכל אחד, ותדעו להריץ 8-10 וורקפלואים בסיסיים.

### שלב 4: שימוש מתקדם ואינטגרציה (שבוע 2+)

**מטרה:** לשלב את BMAD בתהליך העבודה היומיומי ולהשתמש בו בפרויקטים אמיתיים.

**אינטגרציה מלאה:**
- **תחילת פרויקט חדש:** התחילו תמיד עם Samus → Brainstorming → GDD
- **תכנון טכני:** עברו ל-Cloud → Architecture → Solutioning Gate
- **פיתוח אג'ילי:** השתמשו ב-SM (Bob) ל-Sprint Planning, ואז Link לפיתוח Stories
- **Code Reviews:** כל Story שנגמר עובר דרך Link → `*code-review`

**שיתופי פעולה בין אגנטים:**
- Samus (GDD) → Cloud (Architecture) → Link (Implementation)
- Samus (מכניקה חדשה) → Link (מימוש) → TEA (Murat - בדיקות)

**התאמה אישית:**
- התאימו את קבצי ה-config לצוות שלכם
- צרו תבניות GDD מותאמות לסוג המשחקים שלכם
- הגדירו workflows מותאמים אישית (למתקדמים)

**תוצאה:** BMAD משולב בתהליך העבודה שלכם, אתם משתמשים בו יום-יום, וחוסכים שעות רבות בתיעוד ותכנון.

**מה הוא עושה:**

Cloud Dragonborn הוא ארכיטקט מערכות משחק ראשי עם 20+ שנות ניסיון ו-30+ משחקים שהושקו. הוא מתמחה בארכיטקטורת Multiplayer מבוזרת, עיצוב מנועי משחק, רשתות (Networking), תשתית ענן ואופטימיזציה ספציפית לפלטפורמות (PC, Console, Mobile). Cloud מאמין בעיקרון "דחיית החלטות" - לקבל החלטות ארכיטקטורליות רק כאשר יש מספיק מידע. הוא פועל דרך חשיבה תיעוד-ראשון וניתוח שיטתי, מתוך אמונה ששעות של תכנון ארכיטקטורלי חוסכות שבועות של רפקטורינג. הוא מדגיש פשטות כתחכום האולטימטיבי - Scale Adaptive Architecture.

**מתי להשתמש:**
- כאשר צריך לתכנן ארכיטכטורה טכנית למשחק (אחרי GDD מוכן)
- כאשר צריך לעשות החלטות טכנולוגיות גדולות (Unity vs Unreal? Client-Server vs P2P?)
- כאשר המשחק זקוק ל-Multiplayer או Networking מורכב
- כאשר צריך לבצע Solutioning Gate Check - ולידציה שהתכנון הושלם ואפשר לעבור למימוש
- כאשר הפרויקט יוצא ממסלול וצריך Course Correction טכני
- כאשר צריך ייעוץ על ביצועים, אופטימיזציה או Infrastructure

**וורקפלואים עיקריים:**
- `*workflow-status` - בדיקת מצב התכנון הטכני
- `*create-architecture` - יצירת ארכיטקטורה מותאמת סקאלה
- `*solutioning-gate-check` - ולידציה שפאזת Solutioning הושלמה
- `*correct-course` - ניתוח ותיקון מסלול כאשר הפרויקט יוצא מהכיוון

**דוגמה מעשית:**
אתם מפתחים משחק Multiplayer Battle Royale ל-100 שחקנים בו-זמנית. טענו את Cloud (`@game-architect hi`), וודאו שיש GDD מוכן, והריצו `*create-architecture`. Cloud יחקור: Platform? Scale? Network Model (Client-Server/P2P)? Server Architecture (Dedicated/Cloud)? הוא יציע ארכיטקטורה מפורטת: Client (Unity + Client Prediction + Server Reconciliation), Server (C++/Go Dedicated Servers + Matchmaking Service + PostgreSQL + Redis), Network (UDP לגיים + TCP לאירועים), Scalability (Kubernetes + Horizontal Scaling + Load Balancer), Performance Targets (60Hz Tick Rate, <100ms Latency). לאחר מכן, הריצו `*solutioning-gate-check` - Cloud יאשר שהכל מוכן למעבר למימוש.

וורקפלואו מובנה להובלת תהליך Brainstorming יצירתי למשחק חדש. Samus Shepard מוביל אתכם דרך שאלות ממוקדות שבוחנות את הרעיון מכל הזוויות: מהי הפנטזיה המרכזית של השחקן? מה ה-Core Loop (מה השחקן עושה שוב ושוב)? מהן המכניקות הראשוניות? מהם משחקים דומים שמהם אפשר ללמוד? מה הרגש הרצוי?

**קלט:** רעיון ראשוני למשחק (יכול להיות מעורפל).

**פלט:** מסמך Brainstorming מקיף הכולל: High Concept (משפט אחד שמתאר את המשחק), Core Gameplay Loop, רשימת מכניקות ראשוניות, Target Audience, Unique Selling Points, משחקי השראה והשוואה, רשימת שאלות פתוחות לחקירה נוספת.

**מתי להשתמש:** תמיד בהתחלת פרויקט משחק חדש - לפני Game Brief ולפני GDD. זה נקודת הפתיחה שממסגרת את החזון.

**דוגמה:** `@game-designer hi` → `*brainstorm-game` → "רוצה משחק פאזל עם ניהול זמן"

### gdd - יצירת Game Design Document

**מה זה עושה:**
יצירת מסמך עיצוב משחק (GDD) מקיף ומפורט שמשמש כמפת דרכים לכל הצוות. ה-GDD מכיל את כל המידע שצריך כדי לממש את המשחק: Game Overview & Vision, Core Mechanics (מכניקות מפורטות), Progression System (איך השחקן מתקדם), Level Design Guidelines, UI/UX Flow, Economy & Balance (אם רלוונטי), Technical Considerations ו-Scope.

**קלט:** Game Brief (אם קיים) או תוצאות Brainstorming. אם אין, Samus יעזור ליצור תיאור בסיסי.

**פלט:** GDD מקיף במבנה מקצועי, מוכן לשיתוף עם הצוות.

**מתי להשתמש:** אחרי Brainstorming או Game Brief, ולפני תחילת הפיתוח. GDD הוא מסמך חי שמתעדכן כל הזמן בהתאם למשוב ו-Playtesting.

**דוגמה:** `@game-designer hi` → `*gdd`

**פלט:** Narrative Design Document מפורט.

**מתי להשתמש:** אחרי GDD מוכן, רק במשחקים שבהם הנרטיב הוא חלק מרכזי מהחוויה (RPG, Adventure, Story-Driven Games).

### תרחיש 3: מימוש מכניקת Gameplay חדשה

**בעיה:** יש Story במערכת - "מימוש מערכת Dash (זינוק) לדמות השחקן" - וצריך לממש אותו עם קוד ובדיקות.

**פתרון (צעד אחר צעד):**

1. **הכינו Story Context:** וודאו שיש XML/YAML עם: תיאור, Acceptance Criteria, טכנולוגיות (למשל Unity/C#)
2. **טענו את Link Freeman:** `@game-dev hi`
3. **בדקו Stories ממתינות:** הריצו `*workflow-status` - Link יציג רשימת Stories
4. **הריצו מימוש:** `*develop-story` והעבירו את Story Context
5. **Link קורא קריטריונים:**
   - [ ] שחקן יכול ללחוץ Shift למימוש Dash
   - [ ] Dash מזיז את השחקן 10 יחידות בכיוון המבט
   - [ ] Cooldown של 2 שניות בין Dashes
   - [ ] אנימציית Dash משתנה ל-Trail Effect
   - [ ] ביצועים: 60fps ללא ירידות
6. **Link כותב קוד:**
   ```csharp
   public class PlayerController : MonoBehaviour {
       [SerializeField] private float dashDistance = 10f;
       [SerializeField] private float dashCooldown = 2f;
       private float lastDashTime = -999f;
       
       void Update() {
           if (Input.GetKeyDown(KeyCode.LeftShift) && CanDash()) {
               Dash();
           }
       }
       
       bool CanDash() => Time.time >= lastDashTime + dashCooldown;
       
       void Dash() {
           Vector3 dashDirection = transform.forward * dashDistance;
           transform.position += dashDirection;
           lastDashTime = Time.time;
           animator.SetTrigger("Dash");
       }
   }
   ```
7. **Link כותב Tests:**
   - TestCannotDashDuringCooldown()
   - TestDashMovesPlayerCorrectDistance()
8. **Link מריץ בדיקות:** כל הבדיקות עוברות 100%
9. **Code Review:** הריצו `*code-review` - Link מוודא: קוד נקי, קריטריונים מולאו, ביצועים טובים
10. **סגירה:** `*story-done` - Story מסומן כ-Done

**תוצאה:** מכניקת Dash מלאה, נבדקת ומאופטמת, מוכנה ל-Production.

**חיסכון בזמן:** 4-6 שעות עבודה → 1-2 שעות.

## 🔧 התאמה אישית למשחקים

BMAD מאפשר התאמות אישיות נרחבות לתהליכי פיתוח משחקים. הנה האפשרויות החשובות ביותר:

**קבצי קונפיגורציה:**
כל אגנט משחק מגיע עם קובץ YAML שניתן לערוך. למשל, `bmad/bmm/agents/game-designer.agent.yaml` מכיל הגדרות של Samus. אתם יכולים לשנות את סגנון התקשורת, להוסיף workflows מותאמים אישית, ולהגדיר תבניות GDD ספציפיות לז'אנר שלכם.

**תבניות GDD מותאמות:**
הקובץ `bmad/bmm/workflows/2-plan-workflows/gdd/gdd-template.md` הוא התבנית שSamus משתמש בה. אתם יכולים לערוך אותו כדי להתאים ז'אנרים ספציפיים - למשל, תבנית מותאמת ל-Platformers, RPGs או Strategy Games.

**Workflow Paths מותאמים:**
במערכת BMAD ישנו מושג "Workflow Paths" - מסלולי עבודה שונים לפרויקטים שונים. עבור משחקים, יש path מיוחד (`workflow-status/paths/game-design.yaml`) שמתאים לפיתוח משחקים. אתם יכולים ליצור paths נוספים לסוגי משחקים שונים.

**הגדרות צוות:**
אם אתם צוות גדול, שנו את `bmad/bmm/teams/team-gamedev.yaml` כדי להגדיר מי אחראי על מה, איך שולחים notifications וכו'.

**שפה ותקשורת:**
ערכו את `bmad/core/config.yaml` כדי לשנות את שפת התקשורת (עברית/אנגלית), תיקיית פלט לתוצאות, ושם המשתמש. אגנטי BMAD יתאימו עצמם לפי ההגדרות.

**Custom Agents (מתקדם):**
מפתחים מתקדמים יכולים ליצור אגנטים חדשים בפורמט Markdown/YAML. למשל, "Monetization Designer" לעיצוב מערכות כלכליות, או "Level Designer" מיוחד לעיצוב שלבים.

**טיפ:** התחילו עם ההגדרות הדיפולטיביות, והתאימו בהדרגה כשאתם מבינים טוב יותר את צרכי הצוות שלכם.

| **תוצאות לא נשמרות** | בדקו את `output_folder` ב-config.yaml. וודאו שהתיקייה קיימת ויש הרשאות כתיבה |
| **אגנט לא מבין בקשה** | נסו לנסח מחדש בצורה ברורה יותר. השתמשו בפקודות מהתפריט (`*command`) במקום טקסט חופשי |
