## 🎯 למי מדריך זה מיועד

### תיאור התפקיד

מדריך זה נכתב במיוחד עבור **מנהלי מוצר, אנליסטים עסקיים ו-Product Owners** שאחראים על הגדרת חזון המוצר, מחקר שוק, תעדוף פיצ'רים, ויצירת מסמכי דרישות.

אם אתם אלו שצריכים:
- להבין מה השוק צריך ומה המתחרים עושים
- לתרגם צרכים עסקיים ל-PRD או User Stories
- לתעדף פיצ'רים ולהחליט מה נכנס ל-MVP
- לנהל את הוויזיה האסטרטגית של המוצר
- לתקשר בין צוותים טכניים לבין בעלי עניין עסקיים

**זה המדריך בשבילכם!**

### רקע נדרש

- **ניסיון בניהול מוצר או ניתוח עסקי** - לפחות 6 חודשים
- **הבנה בסיסית של תהליכי פיתוח תוכנה** - Agile, Scrum, Epics, Stories
- **יכולת עבודה עם AI IDE** - Cursor, Claude Code, Windsurf או דומיהם
- **אנגלית ברמה בסיסית** - חלק מהמסמכים עשויים להיות באנגלית

### מה תשיגו עם BMAD

אחרי שתשלטו ב-BMAD-METHOD, תוכלו:

✅ **ליצור PRD מקצועי תוך שעות במקום ימים** - איכות גבוהה, מבנה עקבי, בלי להחסיר פרטים קריטיים

✅ **לבצע מחקר שוק מעמיק במהירות** - ניתוח מתחרים, זיהוי הזדמנויות, תובנות מבוססות נתונים

✅ **לפרק פיצ'רים ל-User Stories מדויקים** - עם קריטריוני קבלה, תיעדוף וזמני פיתוח

✅ **לעבוד עם צוות AI וירטואלי** - אנליסטית, מעצב UX, Tech Writer - כולם זמינים תמיד

✅ **לשמור על עקביות ואיכות** - כל מסמך עוקב אחר Best Practices מוכחות

✅ **לחסוך 50-70% מזמן התכנון** - פחות פגישות, פחות איטרציות, יותר ביצוע

#### מה ללמוד

**קראו את המסמכים הבסיסיים:**
1. **התחלה מהירה לכולם** (`00-התחלה-מהירה-לכולם.md`) - 20 דקות
   - הבנה בסיסית של BMAD
   - מבנה המערכת
   - הפילוסופיה מאחורי הגישה

2. **מילון מונחים מלא** (`01-מילון-מונחים-מלא.md`) - 30 דקות
   - אגנט, וורקפלואו, מודול
   - PRD, Epic, Story, Sprint
   - Phase, Track, Level

3. **מדריך אגנטים - חלק 1א** (`02-agents-part-1a-intro-pm-analyst.md`) - 45 דקות
   - היכרות עם John (PM) ו-Mary (Analyst)
   - מתי להשתמש בכל אגנט
   - דוגמאות מעשיות

#### תרגול ראשון

**צפו בתהליך פשוט** - עדיף לראות איך זה עובד לפני שמתקינים:

1. טענו אגנט פשוט:
   ```
   @analyst hi
   ```

2. בקשו הסבר על איך היא עובדת:
   ```
   ספרי לי מה את יכולה לעשות בשבילי
   ```

3. נסו להבין את התפריט שהיא מציגה - אל תריצו שום דבר, רק התרגלו לממשק

#### נקודות המפתח להבין

- **אגנטים** הם מומחי AI עם אישיות ותחום מומחיות ייחודי
- **וורקפלואים** הם תהליכי עבודה מובנים שמדריכים אותך צעד אחר צעד
- **התפריט** הוא נקודת הכניסה לכל הפעולות
- BMAD **לא מחליפה אותך** - היא **משפרת** אותך ומאיצה את העבודה

### שלב 2: התקנה והגדרה ראשונית (שבוע 1-2)

**משך זמן:** 2-3 שעות  
**יעד:** להתקין את BMAD ולהגדיר אותה לצרכים שלכם.

#### התקנת BMAD

**אפשרות 1: התקנה מהירה דרך npx (מומלץ למנהלי מוצר)**
```bash
npx @bmad/cli install
```

עקבו אחרי ההנחיות:
1. בחרו את ה-IDE שלכם (Cursor, Claude Code, וכו')
2. בחרו מודולים: **BMM** (BMAD Method Module) - חובה!
3. הגדירו תיקיית פלט למסמכים
4. הגדירו שפת תקשורת (עברית מומלץ)

**אפשרות 2: התקנה ידנית**
- שימושי אם אתם רוצים שליטה מלאה או עובדים עם מערכת ניהול גרסאות מסובכת
- עקבו אחרי המדריך: `04-התקנה-והגדרה-מתקדמת.md`

#### הגדרת קונפיגורציה אישית

ערכו את הקובץ `bmad/bmm/config.yaml`:

```yaml
# קונפיגורציה למנהל מוצר
user_name: "השם שלכם"
communication_language: "Hebrew"  # עברית לתקשורת
document_output_language: "Hebrew"  # עברית למסמכים (או English לצוותים גלובליים)
output_folder: "{project-root}/product-docs"  # תיקייה ייעודית למסמכי מוצר
user_skill_level: "intermediate"  # או beginner / advanced

# הגדרות למנהלי מוצר
default_prd_format: "comprehensive"  # פורמט PRD מקיף (comprehensive / lean)
include_competitive_analysis: true
include_user_research: true
mvp_focus: true  # דגש על MVP
```

#### וידוא התקנה

1. פתחו צ'אט חדש ב-IDE שלכם
2. טענו אגנט:
   ```
   @pm hi
   ```
3. וודאו שהתפריט מופיע עם כל האפשרויות
4. נסו פקודה פשוטה:
   ```
   *workflow-status
   ```
   (אם זה אומר "No workflow status found" - זה תקין! זה אומר שהמערכת עובדת)

### שלב 4: שימוש מתקדם (שבוע 4-8)

**משך זמן:** זמן רציף - שימוש בפרויקטים אמיתיים  
**יעד:** לשלב BMAD בעבודה היומיומית ולהשתמש בפיצ'רים מתקדמים.

#### התאמות אישיות

**צרו תבניות מותאמות אישית:**
- ערכו את `prd-template.md` לפי הסטנדרטים של החברה שלכם
- התאימו את `product-brief/template.md` לשפה העסקית שלכם
- הוסיפו שדות חובה ספציפיים לארגון

**דוגמה:**
```markdown
## Business Impact (חובה בארגון שלנו)
- Revenue Impact: [estimated]
- User Impact: [metrics]
- Strategic Alignment: [OKR link]
```

#### שילוב עם כלים קיימים

**Jira / Linear / Azure DevOps:**
- השתמשו ב-BMAD ליצירת Stories
- העתיקו לכלי ניהול המשימות שלכם
- שמרו קישור חזרה למסמכי BMAD ב-Description

**Notion / Confluence:**
- שמרו את מסמכי BMAD (PRD, Brief, Research) שם
- השתמשו ב-BMAD כמנוע יצירה, Notion/Confluence כמקום אחסון ושיתוף

**Figma:**
- העבירו את Product Brief ל-Sally (UX Designer)
- קבלו הצעות UX מבוססות מחקר
- העבירו ל-Figma למימוש ויזואלי

#### תרחישים מורכבים

**תרחיש 3: תעדוף ותכנון רבעוני**
```

2. @pm *product-brief  → מה היעדים לרבעון?
```

## 🤖 האג'נטים שלך - מדריך מקוצר

### John (PM) - השותף האסטרטגי שלך

**מה הוא עושה:**

John הוא מנהל מוצר אסטרטגי עם חשיבה אנליטית חדה. הוא מתמחה ביצירת מסמכי דרישות (PRD, Tech Spec), פירוק לאפיקים וסיפורים, ותעדוף פיצ'רים. John לא מקבל תשובות מעורפלות - הוא ידחוף אותך להבין את ה"למה" האמיתי מאחורי כל דרישה. הוא עובד עם מתודולוגיה ברורה, שואל שאלות חודרניות, ומייצר מסמכים מקצועיים לפי תבניות מוכחות. John מבין את ההבדל בין MVP לבין "nice to have", והוא ידאג שהמוצר שלך יישאר ממוקד ובר-ביצוע.

**מתי תשתמש בו:**

1. **יצירת PRD** - כאשר אתה מתכנן מוצר גדול או פיצ'ר מורכב (Level 2-4)
2. **יצירת Tech Spec** - לפרויקטים קטנים-בינוניים (Level 0-2) שלא צריכים PRD מלא
3. **פירוק ל-Epics ו-Stories** - אחרי יצירת PRD, כדי להכין לספרינטים
4. **תעדוף פיצ'רים** - כאשר יש לך רשימה ארוכה וצריך להחליט מה נכנס ל-MVP
5. **Correct Course** - כאשר הפרויקט יוצא ממסלול וצריך ניתוח ותיקון

**איך לטעון אותו:**

```bash
# דרך 1: טעינה עם תפריט
@pm hi

# דרך 2: טעינה והפעלת וורקפלואו ישירות
@pm *create-prd

# דרך 3: שימוש בנתיב מלא (במקרה שצריך)
@pm /bmad:bmm:workflows:prd
```

**דוגמה מעשית:**

**תרחיש:** אתה רוצה לבנות מערכת Onboarding חדשה למוצר SaaS קיים.

```
אתה: @pm hi
John: "היי! אני John, מנהל המוצר. מה העבודה היום?"

אתה: *create-prd
John: "מעולה! בואו ניצור PRD מקצועי. ספר לי קצת על המוצר..."

John: "מי המשתמשים שיעברו את ה-Onboarding?"
```

### prd - יצירת מסמך דרישות מוצר

**מטרת הוורקפלואו:**

וורקפלואו `prd` (Product Requirements Document) הוא הכלי המרכזי ביותר למנהלי מוצר. הוא מנחה אותך דרך תהליך אינטראקטיבי מובנה ליצירת מסמך דרישות מקצועי ומקיף. הוורקפלואו מבוסס על Intent-Driven Planning - הוא מתאים את עצמו לסוג המוצר, המורכבות, וההקשר העסקי. ה-PRD שנוצר כולל: חזון מוצר, דרישות פונקציונליות, קריטריוני הצלחה, אילוצים, סיכונים, ופירוט טכני מדויק. זהו המסמך שמעביר מחזון לביצוע - הצוות הטכני מקבל אותו ויודע בדיוק מה לבנות.

**מתי להשתמש:**

✅ כאשר מתכננים מוצר חדש או פיצ'ר גדול (Level 2-4)  
✅ כאשר יש צורך במסמך מקיף לבעלי עניין רבים  
✅ כאשר הפרויקט מורכב ודורש תכנון מפורט  
✅ כאשר עוקבים אחרי BMad Method או Enterprise Method track  
❌ לא משתמשים ב-PRD לפרויקטים קטנים - במקום זאת השתמשו ב-`tech-spec`

**תשומות נדרשות:**

לפני שמתחילים את ה-PRD, כדאי להכין:
- **Product Brief** (אופציונלי אך מומלץ מאוד) - חזון ברמה גבוהה
- **Market Research** (אופציונלי) - תובנות שוק ומתחרים
- **Project Documentation** (ל-Brownfield) - תיעוד הקוד הקיים
- **הבנה ברורה** של: קהל היעד, הבעיה שנפתרת, יעדי עסק

### שאלות על PRD ו-Tech Specs

**ש6: מתי להשתמש ב-PRD ומתי ב-Tech Spec?**

**ת:** 

**PRD (Product Requirements Document):**
- לפרויקטים גדולים (**Level 2-4**)
- למוצרים מלאים או פיצ'רים מורכבים
- כשיש Stakeholders מרובים
- כשצריך ניתוח עסקי מעמיק
- **זמן:** 2-5 ימים
- **אורך:** 20-50 עמודים

**Tech Spec (Technical Specification):**
- לפרויקטים קטנים-בינוניים (**Level 0-2**)
- לפיצ'רים ממוקדים
- כשהדרישות ברורות
- כשהצוות קטן וזריז
- **זמן:** 2-6 שעות
- **אורך:** 5-15 עמודים

**דוגמאות:**
- מערכת CRM מלאה → **PRD**
- הוספת פיצ'ר "Export to CSV" → **Tech Spec**
- אפליקציה חדשה מאפס → **PRD**
- שיפור UX בדף קיים → **Tech Spec**

**כלל אצבע:** ספק? התחל Tech Spec. אם זה מתגלה מורכב, עבור ל-PRD.

**ש11: מה ההבדל בין PRD ל-Tech Spec? מתי משתמשים בכל אחד?**

**ת:** זו שאלה קריטית!

| | PRD | Tech Spec |
|---|-----|-----------|
| **לאיזה פרויקטים** | גדולים ומורכבים (Level 2-4) | קטנים-בינוניים (Level 0-2) |
| **אורך** | 15-40 עמודים | 5-12 עמודים |
| **פוקוס** | אסטרטגיה, משתמשים, שוק | פונקציונליות, מימוש טכני |
| **קהל יעד** | Stakeholders, Executive, צוות | צוות הפיתוח |
| **כולל** | Vision, Market, Competitors, Epics | Requirements, Architecture, Tasks |
| **זמן יצירה** | 3-8 שעות | 1-3 שעות |
| **מתי** | מוצר חדש, פיצ'ר משמעותי | פיצ'ר בודד, שיפור קיים |

**דוגמאות:**

**צריך PRD:**
- מוצר SaaS חדש מאפס
- Redesign מלא של אפליקציה
- פיצ'ר גדול (marketplace, payment system)

**צריך Tech Spec:**
- הוספת Export to PDF
- מערכת Notifications
- שינוי UI של דף אחד

**כלל אצבע:**
- אם יש Stakeholders חיצוניים → PRD
- אם רק הצוות צריך לדעת → Tech Spec
- אם לא בטוח → התחל Tech Spec, אפשר להרחיב אחר כך

**ש12: איך עובדים עם BMAD במתודולוגיית Agile/Scrum?**

**לפני תחילת Epic:**
```
Sprint -1: @pm *create-prd  → PRD מלא
Sprint 0:  @pm *create-epics-and-stories  → פירוק לסיפורים
Sprint 0:  @sm *sprint-planning  → תכנון 2-3 ספרינטים
```

**במהלך ספרינטים:**
```
Sprint N: 
  - @dev *dev-story  (לכל Story)
  - @tea *test-story  (לכל Story)
  - @pm *workflow-status  (בדיקת התקדמות)
```

**בסוף Epic:**
```
@pm *validate-prd  → וידוא שהכל מושלם
@tech-writer *document-feature  → תיעוד למשתמשים
```

1. **יצרו Epics ב-BMAD** → העתיקו ל-Jira

   ```

   ```

**Ceremonies:**
- **Sprint Planning:** השתמשו ב-Stories שיצרתם
- **Daily Standup:** עדכנו `workflow-status.yaml`
- **Sprint Review:** הראו מה הושלם לפי ה-PRD

**ש23: איך לומדים מפרויקטים שהסתיימו (Retrospective)?**

**בסוף כל פרויקט:**

**1. תעדו מה קרה:**
```markdown

- Planned: 12 weeks
- Actual: 16 weeks (+33%)

## What Went Well ✅
- PRD היה ברור וקומפרטי
- Epics/Stories היו מדויקים
- פחות Rework מהרגיל (10% vs 25% average)

## What Went Wrong ❌
- לא צפינו integration complexity עם API חיצוני
- MVP Scope גדל באמצע (Scope Creep)
- חסר תיאום עם צוות Design

## Lessons Learned 📚
1. יש להוסיף "External Dependencies" section ב-PRD
2. לקבוע MVP freeze date
3. לשלב UX Designer מוקדם יותר
```

**2. עדכנו תבניות:**
```markdown

## [NEW] External Dependencies & Integrations

- Name:
- Purpose:
- Risk Level:
- Fallback Plan:
```

```yaml
  - "Are there external dependencies? (APIs, Services)"
```

**5. תרגלו בפרויקט הבא:**
- השתמשו בתבניות המעודכנות
- בדקו אם השיפורים עזרו
- המשיכו לשפר

💡 **תהליך לחזור עליו:** Retrospective → Update Templates → Apply → Measure → Repeat
