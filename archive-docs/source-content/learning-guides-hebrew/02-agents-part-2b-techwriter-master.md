## Paige - כותבת טכנית (Technical Writer) 📚

**שם באנגלית:** Paige  
**תפקיד:** מומחית תיעוד טכני + אוצרת ידע  
**מומחיות:**
- תקני תיעוד (CommonMark, DITA, OpenAPI)
- תיעוד API וחוויית מפתחים (Developer Experience)
- מדריכים טכניים וארכיטקטורה
- Docs-as-Code ותיעוד חי (Living Documentation)
- תרשימי Mermaid ועזרים ויזואליים
- כתיבה ממוקדת משימות (Task-Oriented Writing)

### 🎯 מתי להשתמש ב-Paige

✅ כאשר צריך לתעד פרויקט קיים (Brownfield) - ניתוח מקיף + תיעוד ארכיטקטורה  
✅ כאשר צריך ליצור תיעוד API מקצועי (OpenAPI/Swagger)  
✅ כאשר צריך מדריכי משתמש או מדריכים טכניים  
✅ כאשר רוצים לבדוק ולשפר תיעוד קיים  
✅ כאשר צריכים ליצור תרשימי Mermaid ברורים  
✅ כאשר רוצים להסביר קונספט מורכב בצורה נגישה

### 📊 וורקפלואים עיקריים

1. **document-project** - תיעוד פרויקט מקיף (ניתוח Brownfield, סריקת ארכיטקטורה)
2. **generate-diagram** - יצירת תרשימי Mermaid (ארכיטקטורה, רצף, זרימה, ER, מחלקות)
3. **validate-doc** - בדיקת תיעוד מול תקנים ו-Best Practices
4. **improve-readme** - סקירה ושיפור של קבצי README
5. **explain-concept** - יצירת הסברים טכניים ברורים עם דוגמאות
6. **standards-guide** - הצגת תקני התיעוד של BMAD

**הערה:** מספר וורקפלואים נוספים (create-api-docs, create-architecture-docs, create-user-guide, audit-docs) עדיין בפיתוח.

### 💬 סגנון תקשורת

Paige היא **מורה סבלנית ותומכת** שגורמת לתיעוד להרגיש נגיש ולא מפחיד.

היא משתמשת ב**דוגמאות ואנלוגיות ברורות** כדי להסביר נושאים מורכבים, ומאזנת **דיוק עם נגישות** - יודעת מתי להיות טכנית ומפורטת ומתי לפשט.

היא **מעודדת הרגלי תיעוד טובים** תוך פרגמטיות לגבי מגבלות העולם האמיתי. Paige חוגגת תיעוד מנוסח היטב ועוזרת לשפר כזה שלא ברור, **ללא שיפוט**.

### 🔑 עקרונות עבודה

Paige מאמינה ש**תיעוד זה הוראה** - כל מסמך צריך לעזור למישהו להשיג משימה ספציפית, לא רק לתאר פיצ'רים.

הפילוסופיה שלה מאמצת **בהירות מעל הכל** - היא משתמשת בשפה פשוטה, תוכן מובנה, ועזרים ויזואליים (תרשימי Mermaid) כדי להפוך נושאים מורכבים לנגישים.

היא מתייחסת לתיעוד כ**ארטיפקטים חיים** שמתפתחים עם הקוד, ותומכת ב-**Docs-as-Code** ותחזוקה רציפה במקום יצירה חד-פעמית.

היא פועלת עם **תקנים כבסיס** (CommonMark, OpenAPI, Style Guides) תוך גמישות לצרכי הפרויקט, תמיד נותנת **עדיפות לחוויית הקורא** על פני דבקות נוקשה בכללים.

### 📁 איפה למצוא

קובץ: `bmad/bmm/agents/tech-writer.md`  
מודול: BMM (BMAD Method Module)

### 💡 דוגמה מעשית מלאה

**תרחיש:** יש לך פרויקט Node.js קיים שמבולגן ובלי תיעוד, ואתה רוצה לתעד אותו לפני שמוסיפים פיצ'רים.

**שלבים:**
1. טען את Paige: `@tech-writer hi`
2. הרץ: `*document-project`
   - Paige תסרוק את כל בסיס הקוד
   - תנתח:
     - מבנה התיקיות והקבצים
     - טכנולוגיות וספריות בשימוש
     - נקודות כניסה (Entry Points)
     - APIs ציבוריים ופנימיים
     - תלויות בין מודולים
     - קונפיגורציות ו-Environment Variables
   - תייצר מסמכי תיעוד מקיפים:
     - **PROJECT-OVERVIEW.md:** סקירה כללית של הפרויקט
     - **ARCHITECTURE.md:** ארכיטקטורה עם תרשימי Mermaid
     - **API-REFERENCE.md:** תיעוד כל ה-APIs
     - **DEVELOPER-GUIDE.md:** מדריך למפתחים חדשים
     - **DEPLOYMENT.md:** הוראות פריסה
3. אם רוצה לשפר את ה-README הקיים, הרץ: `*improve-readme`
   - Paige תנתח את ה-README הנוכחי
   - תציע שיפורים:
     - חסרים סעיפים? (Getting Started, Usage, Contributing)
     - לא ברור? איפה?
     - דוגמאות קוד חסרות?
     - לינקים שבורים?
   - תציע גרסה משופרת
4. אם יש קונספט מורכב שצריך הסבר, הרץ: `*explain-concept`
   - בקש: "הסבירי איך עובד ה-authentication flow"
   - Paige תיצור הסבר מובנה עם:
     - תרשים Sequence של הזרימה
     - הסבר שלב אחר שלב
     - דוגמאות קוד
     - מקרי קצה (Edge Cases)
5. אם צריך תרשים, הרץ: `*generate-diagram`
   - Paige תשאל: "איזה סוג תרשים? (flowchart, sequence, class, ER, state, git)"
   - תשאל על התוכן
   - תייצר תרשים Mermaid מקצועי ותקין
6. לפני סיום, הרץ: `*validate-doc` על המסמכים החדשים
   - Paige תבדוק:
     - התאמה לתקן CommonMark
     - Best Practices של כתיבה טכנית
     - בהירות ונגישות
     - תקינות תרשימים
   - תציע שיפורים ספציפיים לפי עדיפות

**תוצאה:** פרויקט מתועד באופן מקצועי עם ארכיטקטורה ברורה, מדריכים ותרשימים שמאפשרים onboarding מהיר של מפתחים חדשים.

### 🤝 עובד טוב עם

- **Analyst (Mary)** - Mary מתעדת פרויקטים קיימים, Paige משפרת ומפרטת את התיעוד
- **Architect (Winston)** - Winston מתכנן ארכיטקטורה, Paige מתעדת אותה
- **PM (John)** - Paige יכולה לתעד PRDs ומפרטים באופן מקצועי
- **DEV (Amelia)** - תיעוד API ומדריכי מפתחים

### ⚠️ טעויות נפוצות

1. **לדלג על document-project בפרויקט קיים** - מוסיפים פיצ'רים בלי להבין מה כבר יש. זה מתכון לבאגים.

2. **לא לשמור תיעוד מעודכן** - יוצרים תיעוד פעם אחת ושוכחים לעדכן. תיעוד ישן גרוע מאין תיעוד.

3. **תרשימים לא תקינים** - ליצור תרשימי Mermaid ידנית ללא בדיקה. Paige תיצור תרשימים תקינים.

4. **לכתוב לעצמכם, לא למשתמש** - תיעוד שמתאר מה הקוד עושה במקום איך להשתמש בו.

5. **README חסר או חלש** - README זה פתח לפרויקט. אם הוא לא טוב, אף אחד לא ישתמש בפרויקט.

### 🎓 טיפים מתקדמים

1. **Docs-as-Code זה הדרך** - שמרו תיעוד בגיט ליד הקוד, תנהלו אותו כמו קוד.

2. **CommonMark הוא חברכם** - Paige קפדנית על התקן. זה מבטיח שהתיעוד יעבוד בכל מקום.

3. **תרשימי Mermaid חוסכים אלפי מילים** - תמונה שווה אלף מילים, תרשים אינטראקטיבי שווה עשרת אלפים.

4. **validate-doc לפני Merge** - תבדקו תיעוד כמו שבודקים קוד. איכות חשובה.

5. **השקיעו ב-README** - זה הדבר הראשון שכולם רואים. הפכו אותו למעולה.

## BMad Master - מתזמר ראשי 🧙

**שם באנגלית:** BMad Master  
**תפקיד:** מבצע משימות ראשי + מומחה BMad + מתזמר  
**מומחיות:**
- ידע מקיף בכל פלטפורמת BMAD וכל המודולים
- ביצוע משימות ישיר וניהול משאבים בזמן ריצה
- תזמור וורקפלואים מורכבים
- מצב "Party Mode" - ריבוי אגנטים בשיח אחד
- מנוע הביצוע הראשי של פעולות BMAD

### 🎯 מתי להשתמש ב-BMad Master

✅ כאשר אתה לא בטוח באיזה אגנט להשתמש - BMad Master ידריך אותך  
✅ כאשר רוצה לראות רשימה של כל המשימות והוורקפלואים הזמינים  
✅ כאשר צריך לתזמר מספר אגנטים יחד ב-Party Mode  
✅ כאשר רוצה מידע כללי על המערכת והסטטוס  
✅ כאשר זה נקודת ההתחלה שלך ואתה צריך הדרכה

### 📊 וורקפלואים עיקריים

1. **list-tasks** - רשימת כל המשימות הזמינות מ-task-manifest.csv
2. **list-workflows** - רשימת כל הוורקפלואים הזמינים מ-workflow-manifest.csv
3. **party-mode** - צ'אט קבוצתי עם כל האגנטים (מצב שיתוף פעולה)

### 💬 סגנון תקשורת

BMad Master מתקשר בצורה **ישירה ומקיפה**, ומתייחס לעצמו **בגוף שלישי**.

התקשורת שלו **ממוקדת ביצוע יעיל של משימות**, מציגה מידע **באופן שיטתי עם רשימות ממוספרות** ויכולת תגובה מיידית לפקודות.

### 🔑 עקרונות עבודה

BMad Master טוען **משאבים בזמן ריצה, לעולם לא מראש** - זה מבטיח שהוא תמיד עובד עם המידע העדכני ביותר.

הוא תמיד מציג **רשימות ממוספרות לבחירות** - זה מקל על הניווט והאינטראקציה.

### 📁 איפה למצוא

קובץ: `bmad/core/agents/bmad-master.md`  
מודול: Core

### 💡 דוגמה מעשית מלאה

**תרחיש:** אתה חדש ב-BMAD ולא בטוח מאיפה להתחיל.

**שלבים:**
1. טען את BMad Master: `@bmad-master hi`
2. BMad Master יקרא את הקונפיגורציה שלך
3. יציג תפריט:
   ```
   1. Show numbered menu (*help)
   2. List Available Tasks (*list-tasks)
   3. List Workflows (*list-workflows)
   4. Group chat with all agents (*party-mode)
   5. Exit with confirmation (*exit)
   ```
4. הרץ: `*list-workflows`
   - BMad Master יציג את כל הוורקפלואים המותקנים
   - מחולק לפי מודולים (BMM, CIS, BMB, Core)
   - עם הסבר קצר על כל אחד
5. הרץ: `*list-tasks`
   - רשימת כל המשימות שאפשר להריץ
   - שימושי לראות מה זמין
6. אם רוצה לעשות דיון עם מספר אגנטים, הרץ: `*party-mode`
   - BMad Master יאסוף מספר אגנטים לצ'אט אחד
   - אתה יכול לשאול שאלות וכולם יענו מנקודת המבט שלהם
   - שימושי לדיונים מורכבים שדורשים נקודות מבט שונות
   - לדוגמה: "איך להטמיע פיצ'ר authentication?"
     - PM יענה מזווית עסקית ודרישות
     - Architect יענה מזווית ארכיטקטורה
     - Security Expert יענה מזווית אבטחה
     - DEV יענה מזווית מימוש

**תוצאה:** הבנה ברורה של מה זמין במערכת ואיך להתחיל.

### 🤝 עובד טוב עם

- **כל האגנטים** - BMad Master הוא נקודת הכניסה והמתזמר הראשי
- במיוחד שימושי כנקודת התחלה למשתמשים חדשים

### ⚠️ טעויות נפוצות

1. **לא להתחיל עם BMad Master** - לקפוץ ישר לאגנט ספציפי בלי להבין את התמונה הכללית.

2. **לא להשתמש ב-list-workflows** - לא לדעת מה זמין. זה כמו לא לקרוא את התפריט במסעדה.

3. **להימנע מ-Party Mode** - זה נראה "משוגע", אבל זה באמת עוצמתי לדיונים מורכבים.

4. **לא לחזור ל-BMad Master כשתקועים** - הוא תמיד יכול להדריך אותך למשאב הנכון.

### 🎓 טיפים מתקדמים

1. **BMad Master הוא המצפן שלך** - כשאתה אבוד, תחזור אליו.

2. **Party Mode לדיונים אסטרטגיים** - כשאתה צריך לקבל החלטה גדולה, תשתמש ב-Party Mode.

3. **list-workflows בתחילת כל פרויקט** - כדי לראות מה רלוונטי לפרויקט הספציפי שלך.

4. **השתמש בו כ-"לובי"** - התחל תמיד עם BMad Master, קבל הכוונה, ואז עבור לאגנט הספציפי.

5. **לא בושה לשאול** - BMad Master לא ישפוט אותך על שאלות "טיפשיות". זה התפקיד שלו לעזור.
