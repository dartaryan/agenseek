## 🌟 מושגי יסוד

### BMAD-METHOD

**מה זה:** מתודולוגיה מקיפה לבניית תוכנה עם עזרת בינה מלאכותית. זה בעצם "מערכת הפעלה" לעבודה עם AI בפיתוח תוכנה.

**דוגמה:** במקום להשתמש ב-ChatGPT בצורה אקראית, BMAD נותן לכם תהליך מובנה עם מומחים מיוחדים (אג'נטים) לכל שלב.

**קיצור:** BMad או BMAD

---

### BMad-CORE (ליבת המערכת)

**מה זה:** הליבה הטכנולוגית שמפעילה את כל מערכת BMAD. זה כמו "מנוע" שמריץ את האג'נטים והוורקפלואים.

**מה זה עושה:**
- מנהל את האג'נטים
- מריץ את הוורקפלואים
- דואג לאינטגרציה עם ה-IDE
- שומר הגדרות אישיות

**דוגמה:** כשאתם טוענים אג'נט, BMad-CORE הוא מי שבעצם "מעיר" אותו לחיים.

---

### אג'נט (Agent)

**מה זה:** בינה מלאכותית מיוחדת שמתמחה בתפקיד ספציפי - כמו מנהל מוצר, ארכיטקט, או מפתח.

**מה הוא יודע לעשות:**
- לדבר איתכם כמו איש מקצוע אמיתי
- להציע פתרונות מבוססי ניסיון
- להריץ וורקפלואים מתוחכמים
- לעזור לכם לקבל החלטות נכונות

**דוגמה:** האג'נט "John" (PM) יודע איך לכתוב דרישות מוצר (PRD) כמו מנהל מוצר ותיק עם 10 שנות ניסיון.

**כמה יש:** 19+ אג'נטים שונים במערכת המלאה.

---

### וורקפלואו (Workflow)

**מה זה:** תהליך עבודה מובנה ומדויק שעוזר לכם להשלים משימה מסוימת שלב אחר שלב.

**איך זה עובד:**
1. אתם מפעילים וורקפלואו (למשל: `*workflow-init`)
2. הוורקפלואו שואל שאלות
3. אתם עונים
4. הוורקפלואו מייצר פלט (מסמך, קוד, החלטה)

**דוגמה:** הוורקפלואו "prd" מנחה אתכם דרך כל השלבים של כתיבת מסמך דרישות מוצר - מחקר, יעדים, תכונות, קריטריוני הצלחה.

**כמה יש:** 50+ וורקפלואים שונים.

---

### מודול (Module)

**מה זה:** חבילה של אג'נטים ווורקפלואים שמתמחים בתחום מסוים.

**המודולים העיקריים:**

🔹 **BMM (BMad Method)** - פיתוח תוכנה מלא (12 אג'נטים, 34 וורקפלואים)  
🔹 **BMB (BMad Builder)** - יצירת אג'נטים ווורקפלואים חדשים  
🔹 **CIS (Creative Intelligence Suite)** - יצירתיות וחדשנות (5 אג'נטים, 5 וורקפלואים)

**דוגמה:** אם אתם מפתחים תוכנה, תתקינו את מודול BMM. אם אתם אנשי שיווק יצירתיים, תתקינו את CIS.

---

### IDE (סביבת פיתוח)

**מה זה:** התוכנה שבה אתם כותבים קוד ועובדים עם BMAD.

**דוגמאות נפוצות:**
- **VS Code** (Visual Studio Code)
- **Cursor** (IDE עם AI מובנה)
- **Claude Code** (IDE חדש מבית Anthropic)

**למה זה חשוב:** BMAD משתלב עם ה-IDE שלכם כך שהאג'נטים יכולים לעבוד ישירות על הקוד.

## 📊 וורקפלואים (Workflows)

### workflow-init (אתחול פרויקט)

**מה זה עושה:** מבין מה הפרויקט שלכם ויוצר קובץ מעקב (workflow status file).

**מתי להשתמש:** **תמיד** בתחילת פרויקט חדש!

**פלט:** קובץ `bmm-workflow-status.yaml` שעוקב אחרי ההתקדמות.

---

### workflow-status (סטטוס פרויקט)

**מה זה עושה:** בודק איפה אתם בתהליך ואומר לכם מה לעשות הבא.

**מתי להשתמש:** כשאתם לא בטוחים מה הצעד הבא.

**דוגמת תשובה:**
```
You're in Phase 2 (Planning). 
Next required step: Run 'prd' workflow to create requirements.
```

---

### prd (Product Requirements Document)

**מה זה עושה:** מדריך אתכם ליצירת מסמך דרישות מוצר מקצועי ומקיף.

**משך זמן:** 1-3 שעות (תלוי בפרויקט)

**פלט:** מסמך PRD מפורט עם Epics מפורקים.

**מתי להשתמש:** פרויקטים בינוניים-גדולים (Level 2+).

---

### tech-spec (מפרט טכני מהיר)

**מה זה עושה:** יוצר מפרט טכני קצר לפרויקטים פשוטים.

**משך זמן:** 15-60 דקות

**פלט:** מסמך Tech-Spec מפורט.

**מתי להשתמש:** באגים, תכונות קטנות, שינויים פשוטים (Level 0-1).

---

### architecture (ארכיטקטורה)

**מה זה עושה:** מתכנן את המבנה הטכני של המערכת.

**משך זמן:** 2-4 שעות

**פלט:** מסמך Architecture מפורט.

**מתי להשתמש:** פרויקטים עם 10+ Stories.

---

**איך למנוע:** השתמשו ב-Fresh Chats, תנו קונטקסט ברור, עבדו עם מסמכים מובנים.

## 🔄 תהליכים ושיטות עבודה

### Scale-Adaptive System (מערכת מותאמת לסקלה)

**מה זה:** BMAD מתאים את עומק התכנון לגודל ומורכבות הפרויקט.

**3 מסלולים (Tracks):**

**1. Quick Flow Track (Level 0-1)**
- באגים ותכונות קטנות
- רק Tech-Spec, בלי PRD
- מהיר וזריז

**2. BMad Method Track (Level 2-3)**
- מוצרים ופלטפורמות
- PRD + Architecture מלא
- תכנון מובנה

**3. Enterprise Method Track (Level 4)**
- מערכות ארגוניות
- PRD + Architecture + Security/DevOps/Test
- תכנון מקיף

**איך בוחרים:** הוורקפלואו `workflow-init` ממליץ ומנחה.

---

### 4 הפאזות (Four Phases)

**Phase 0: Documentation (Brownfield בלבד)**
- תיעוד קוד קיים
- הבנת המערכת הקיימת

### Story-Centric Development (פיתוח ממוקד Stories)

**מה זה:** גישה שבה כל עבודת הפיתוח מתבצעת דרך User Stories קטנים ומנוהלים.

**מחזור חיים של Story:**

```
BACKLOG → DRAFTED → READY → IN PROGRESS → REVIEW → DONE
```

**למה זה טוב:**
- עבודה ממוקדת ובהירה
- AI לא מתבלבל
- מעקב פשוט
- קל לנהל

---

### Epic (אפיק)

**מה זה:** קבוצת User Stories הקשורים לתכונה גדולה אחת.

**דוגמה:**

**Epic:** "מערכת התחברות משתמשים"

**Stories בתוכו:**
- Story 1: יצירת מסך התחברות
- Story 2: אימות מול מסד נתונים
- Story 3: Remember Me functionality
- Story 4: שכחתי סיסמה

**גודל:** בדרך כלל 3-15 Stories.

---

### User Story

**מה זה:** תיאור קצר של תכונה בודדת מנקודת מבט של המשתמש.

**פורמט קלאסי:**
```
As a [role]
I want [feature]
So that [benefit]
```

**דוגמה:**
```
As a registered user
I want to reset my password
So that I can regain access if I forget it
```

**גודל אידאלי:** משימה שניתן לסיים ב-1-3 ימי עבודה.

---

### Acceptance Criteria (קריטריוני קבלה)

**מה זה:** רשימת תנאים ברורים שמגדירים מתי Story נחשב "סיים".

**דוגמה:**
```
✅ משתמש יכול ללחוץ על "שכחתי סיסמה"
✅ מערכת שולחת אימייל עם קישור איפוס
✅ הקישור תקף ל-24 שעות
✅ משתמש יכול להזין סיסמה חדשה
✅ סיסמה חדשה עוברת ולידציה
```

**למה זה חשוב:** אין עוררין מתי העבודה הושלמה.

---

### Definition of Done (DoD) (הגדרת "סיים")

**מה זה:** רשימת משימות שצריך להשלים לפני שמסמנים Story כ-DONE.

**דוגמה:**
```
✅ קוד נכתב
✅ טסטים יוניט עוברים
✅ Code Review הושלם
✅ תיעוד עודכן
✅ Story הועבר ל-DONE בסטטוס
```

### Brownfield Project (פרויקט קיים)

**מה זה:** פרויקט עם קוד קיים (לא מתחילים מאפס).

**האתגר:** צריך להבין מה כבר קיים לפני שמוסיפים תכונות.

**הפתרון ב-BMAD:** הוורקפלואו `document-project` סורק ומתעד את הקוד הקיים.

**הניגוד:** Greenfield Project (פרויקט חדש לגמרי).

---

### Greenfield Project (פרויקט חדש)

**מה זה:** פרויקט שמתחילים מאפס, ללא קוד קיים.

**היתרון:** חופש לעצב הכל מההתחלה.

**האתגר:** צריך לקבל הרבה החלטות טכניות.

## 📁 קבצים ותיקיות

### bmad/ (תיקיית השורש)

**מה זה:** התיקייה הראשית שבה כל קבצי BMAD נמצאים.

**מבנה:**
```
bmad/
├── core/         # ליבה + BMad Master
├── bmm/          # מודול פיתוח
├── bmb/          # מודול Builder
├── cis/          # מודול יצירתיות
└── _cfg/         # הגדרות אישיות
```

---

### bmad/_cfg/ (תיקיית הגדרות)

**מה זה:** תיקייה שבה ההגדרות האישיות שלכם שרודות עדכונים.

**מה יש בה:**
- `agent-manifest.csv` - רשימת כל האג'נטים
- `workflow-manifest.csv` - רשימת כל הוורקפלואים
- `agents/` - קבצי התאמה אישית לאג'נטים

**חשוב:** אל תמחקו תיקייה זו! היא שומרת את ההגדרות שלכם.

---

### config.yaml (קובץ הגדרות)

**מה זה:** קובץ הגדרות ראשי של כל מודול.

**מה יש בו:**
```yaml
user_name: שם המשתמש שלכם
communication_language: עברית או אנגלית
document_output_language: שפת המסמכים
output_folder: תיקיית הפלטים
```

**איפה:** יש אחד בכל מודול (`bmad/core/config.yaml`, `bmad/bmm/config.yaml`, וכו').

---

### bmm-workflow-status.yaml (קובץ מעקב)

**מה זה:** קובץ שעוקב אחרי ההתקדמות שלכם בפרויקט.

**מה יש בו:**
- איזה פאזה אתם נמצאים
- אילו וורקפלואים הושלמו
- מה הצעד הבא

**נוצר על ידי:** הוורקפלואו `workflow-init`.

**איפה:** בתיקיית המסמכים שלכם (בדרך כלל `docs/`).

---

### PRD (Product Requirements Document)

**מה זה:** מסמך דרישות מוצר - מה אנחנו בונים ולמה.

**מה כולל:**
- תיאור המוצר
- יעדים עסקיים
- קהל יעד
- תכונות עיקריות
- קריטריוני הצלחה
- Epics מפורקים

**נוצר על ידי:** הוורקפלואו `prd` עם אג'נט PM.

---

### Architecture Document (מסמך ארכיטקטורה)

**מה זה:** מסמך תכנון טכני - איך אנחנו בונים את זה.

**מה כולל:**
- מבנה המערכת
- טכנולוגיות נבחרות
- API Design
- מסד נתונים
- החלטות אבטחה
- שיקולים טכניים

**נוצר על ידי:** הוורקפלואו `architecture` עם אג'נט Architect.

## 📐 מונחי תכנון ופיתוח

### PRD - Product Requirements Document

ראו למעלה בקטגוריית "קבצים".

---

### GDD - Game Design Document

**מה זה:** מסמך תכנון משחק - גרסת PRD למשחקים.

**מה כולל:**
- Concept וחזון
- מכניקות ליבה
- מערכות משחק
- לולאות התקדמות
- אמנות וסאונד
- מפרט טכני

**נוצר על ידי:** הוורקפלואו `gdd` עם אג'נט Game Designer.

---

### Tech-Spec - Technical Specification

**מה זה:** מפרט טכני מקוצר לשינויים קטנים.

**מה כולל:**
- תיאור הבעיה
- פתרון מוצע
- שינויים נדרשים
- Acceptance Criteria

**נוצר על ידי:** הוורקפלואו `tech-spec` עם אג'נט PM.

---

### MVP - Minimum Viable Product

**מה זה:** הגרסה הבסיסית ביותר של המוצר שעדיין משיגה ערך.

**למה חשוב:** לא בונים הכל מהיום הראשון - מתחילים ב-MVP ומוסיפים אחר כך.

**דוגמה:** 
- ❌ לא MVP: אתר אי-קומרס עם 50 תכונות
- ✅ MVP: רק קטלוג מוצרים + עגלת קניות + תשלום

---

### API - Application Programming Interface

**מה זה:** ממשק תקשורת בין מערכות או חלקי תוכנה.

**דוגמה:** API של התחזית - מערכת שלכם שולחת בקשה ומקבלת נתוני מזג אוויר.

**ב-BMAD:** הארכיטקט מתכנן את ה-APIs במסמך הארכיטקטורה.

---

### CI/CD - Continuous Integration / Continuous Deployment

**מה זה:** תהליכים אוטומטיים שבודקים ומפרסמים קוד.

**CI (Integration):** כל שינוי עובר טסטים אוטומטיים.  
**CD (Deployment):** קוד שעבר טסטים מתפרסם אוטומטית.

**ב-BMAD:** אג'נט TEA עוזר לבנות pipeline של CI/CD.

---

### TDD - Test Driven Development

**מה זה:** גישה שבה כותבים טסט לפני הקוד.

**תהליך:**
1. כתוב טסט (הוא נכשל - אין קוד)
2. כתוב מינימום קוד שיעביר את הטסט
3. שפר את הקוד (Refactor)
4. חזור

**ב-BMAD:** הוורקפלואו `testarch-atdd` מיישם TDD.

## ❓ שאלות נפוצות על מונחים

### "אני מתבלבל בין Workflow ו-Task. מה ההבדל?"

**Workflow:** תהליך מלא שמורכב ממספר שלבים. אינטראקטיבי - שואל שאלות ומדריך אתכם.  
**Task:** פעולה אטומית בודדת שהמערכת מבצעת. לא אינטראקטיבי.

**דוגמה:**  
- Workflow: `prd` - תהליך שלם של יצירת PRD עם שיחה ושאלות  
- Task: `index-docs` - רק מייצר אינדקס של מסמכים

---

### "מה ההבדל בין Agent ו-LLM רגיל?"

**LLM רגיל (כמו ChatGPT):** בינה מלאכותית כללית ללא התמחות.

**Agent:** LLM עם:
- התמחות בתפקיד מסוים
- אישיות ייחודית
- גישה לוורקפלואים
- הקשר ספציפי למשימה

**דוגמה:** במקום לשאול את ChatGPT "תכתוב לי PRD", אתם משוחחים עם PM Agent שיודע בדיוק איך לכתוב PRD מקצועי ויכוון אתכם בתהליך.

---

### "למה יש כל כך הרבה אג'נטים? לא מספיק אחד?"

כל אג'נט מתמחה בתחום שלו:

- **PM** חושב כמו איש עסקים - מה לבנות ולמה
- **Architect** חושב כמו טכנאי - איך לבנות
- **DEV** חושב כמו מבצע - הקוד הספציפי

**אותו AI, תפקידים שונים = תוצאות טובות יותר.**
