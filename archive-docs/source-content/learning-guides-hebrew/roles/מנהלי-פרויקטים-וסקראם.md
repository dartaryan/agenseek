## 🎯 למי מדריך זה מיועד

### Scrum Masters 🏃
- אחראים על תהליכי Agile והנחיית צוותי פיתוח
- מנהלים ספרינטים, רטרוספקטיבות, ו-Daily Standups
- מכינים Stories ומבטיחים שהצוות עובד ביעילות מקסימלית

### Agile Coaches 🎯
- מלווים צוותים ביישום Agile נכון
- מטפלים בחסמים ארגוניים ומשפרים תהליכים
- מדריכים מנהלים ומפתחים בשיטות עבודה זריזות

### Technical Project Managers 📊
- מנהלים פרויקטי תוכנה מורכבים
- מתאמים בין צוותים טכניים ובעלי עניין עסקיים
- אחראים על תכנון, ביצוע, ומעקב אחר פרויקטים

### מה תשיגו עם BMAD-METHOD?

✅ **ניהול ספרינטים אוטומטי:** מעקב אחר Stories, Epics, וסטטוס ספרינטים ללא עבודה ידנית  
✅ **הכנת Stories מהירה:** יצירת User Stories מפורטים ומוכנים לפיתוח בדקות  
✅ **רטרוספקטיבות מקצועיות:** הנחיית רטרו אינטראקטיבית עם כל הצוות  
✅ **תיקון מסלול מבוקר:** טיפול בשינויים ובעיות במהלך הספרינט בצורה מובנית  
✅ **תיעוד אוטומטי:** כל המידע מתועד ונגיש, אין צורך בעבודת תיעוד ידנית  
✅ **צמצום פגישות:** מידע זמין לכולם, פחות פגישות תיאום  
✅ **איכות גבוהה יותר:** Story Context מבטיח שהמפתחים מבינים את הדרישות

## 🚀 מסלול למידה מהיר

### שלב 1: התחלה (יום 1)

**מטרה:** להבין איך BMAD משנה את אופן ניהול הספרינטים שלכם.

BMAD-METHOD הוא מערכת ניהול ספרינטים אוטומטית המבוססת על AI Agents. במקום לעדכן ידנית Excel/Jira ולרוץ אחרי מפתחים לקבל סטטוס, BMAD מנהל את הכל אוטומטית.

**המושגים המרכזיים:**

- **Agent:** עוזר AI עם תפקיד ספציפי (SM, PM, DEV, וכו')
- **Workflow:** תהליך מובנה שמריץ משימה מסוימת (sprint-planning, create-story, retrospective)
- **Story Context:** XML מרוכז עם כל המידע שמפתח צריך לממש Story
- **sprint-status.yaml:** קובץ מעקב מרכזי עם סטטוס כל ה-Stories וה-Epics

**התהליך הבסיסי:**

1. PM יוצר PRD (Product Requirements Document) עם כל הדרישות
2. Architect מתכנן Architecture טכנית
3. PM יוצר Epics ו-Stories מה-PRD
4. **אתה (SM)** מריץ `sprint-planning` ליצירת sprint-status.yaml
5. **אתה** מריץ `create-story` ליצירת Story Drafts מפורטים
6. **אתה** מריץ `story-context` להכנת Story Context XML למפתחים
7. מפתחים מממשים את ה-Stories עם DEV Agent
8. **אתה** מריץ `retrospective` בסיום Epic

**למה זה משנה הכל:**

- אין יותר Excel מסובך או Jira שצריך עדכון ידני
- המפתחים מקבלים את כל המידע שהם צריכים, אין שאלות אינסופיות
- הרטרוספקטיבה היא ישיבת צוות אמיתית, לא סתם "מה היה טוב/רע"
- תיקון מסלול במהלך הספרינט נעשה בצורה מובנית ומבוקרת

**למידה יום 1:**
- קרא את `00-התחלה-מהירה-לכולם.md` (30 דקות)
- קרא את `01-מילון-מונחים-מלא.md` - סעיפי Agile (20 דקות)
- קרא את `02-agents-part-1b-architect-sm-dev.md` - חלק Bob (SM) (30 דקות)

### שלב 2: התקנה (יום 1-2)

**מטרה:** להתקין ולהגדיר את BMAD לעבודה מול הצוות שלך.

**התקנה בסיסית (20 דקות):**

```bash
# התקן את BMAD
npx bmad-cli install

# התקן את מודול BMM (BMAD Method Module)
npx bmad-cli install bmm
```

**הגדרת קונפיגורציה (10 דקות):**

ערוך את `bmad/bmm/config.yaml`:

```yaml
user_name: "השם שלך"
communication_language: "Hebrew"
document_output_language: "Hebrew"
output_folder: "{project-root}/docs"
dev_story_location: "{project-root}/stories"
```

**הגדרות ספציפיות ל-SM:**

- `dev_story_location`: היכן ה-Stories נשמרים (בדרך כלל `stories/` או `docs/stories/`)
- `output_folder`: היכן מסמכי הפרויקט (PRD, Architecture) נשמרים

**אימות התקנה (5 דקות):**

1. טען את BMad Master: `@bmad-master hi`
2. בדוק שהוא עולה בהצלחה ומזהה את השם שלך
3. הרץ: `*list-workflows` לראות את כל הוורקפלואים הזמינים

**מידע נוסף:** ראה `04-התקנה-והגדרה-מתקדמת.md` להתקנה מתקדמת.

### שלב 3: אג'נטים ווורקפלואים (שבוע 1)

**מטרה:** לשלוט באג'נטים והוורקפלואים המרכזיים לעבודתך היום-יומית.

**האג'נט הראשי שלך: Bob (SM)**

Bob הוא ה-Scrum Master הטכני שלך. הוא מומחה בהכנת Stories ותיאום ספרינטים.

**טען את Bob:**
```
@sm hi
```

Bob יציג לך תפריט עם כל הפקודות שלו:
- `*sprint-planning` - ניהול sprint-status.yaml
- `*create-story` - יצירת Story חדש
- `*story-context` - הכנת Story Context XML
- `*retrospective` - הנחיית רטרוספקטיבה
- `*correct-course` - תיקון מסלול

**5 הוורקפלואים החשובים ביותר:**

1. **workflow-status** - מה המצב ומה לעשות הלאה? (התחל כאן תמיד!)
2. **sprint-planning** - יצירת/עדכון sprint-status.yaml מקבצי Epic
3. **create-story** - יצירת Story Draft חדש מתוך Epic
4. **story-context** - הכנת Story Context XML למפתחים
5. **retrospective** - הנחיית רטרו אחרי Epic

**תרגול יום-יומי (שבוע 1):**

**יום 1-2:** תרגל להריץ `workflow-status` ו-`sprint-planning`
- פתח פרויקט קיים (או צור חדש)
- הרץ workflow-status לראות איפה הפרויקט נמצא
- הרץ sprint-planning ובדוק את sprint-status.yaml שנוצר

**יום 3-4:** תרגל `create-story`
- בחר Epic קיים
- הרץ create-story ליצירת Story מפורט
- בדוק את ה-Story Draft שנוצר - האם הוא מכיל Acceptance Criteria ברורים?

**יום 5:** תרגל `story-context`
- קח Story שיצרת
- הרץ story-context לבניית ה-XML
- בדוק את ה-Story Context - האם יש בו את כל המידע?

**סימנים שאתה מוכן לשלב 4:**
- ✅ יצרת לפחות 3 Stories בהצלחה
- ✅ הרצת sprint-planning וקראת את sprint-status.yaml
- ✅ יצרת Story Context אחד והבנת מה יש בו
- ✅ אתה מרגיש בנוח עם Bob ואתה יודע איפה למצוא כל פקודה

### שלב 4: שימוש מתקדם (שבוע 2+)

**מטרה:** לעבוד עם כל האג'נטים ביחד ולהריץ ספרינטים מלאים.

**עבודה עם אג'נטים נוספים:**

1. **PM (John)** - עבודה עם PRD ו-Epics
   - טען: `@pm hi`
   - הרץ: `*create-prd` ליצירת PRD חדש
   - הרץ: `*create-epics-and-stories` לפירוק PRD ל-Epics

2. **BMad Master** - תיאום כללי ומעקב
   - טען: `@bmad-master hi`
   - הרץ: `*list-workflows` לראות את כל הוורקפלואים
   - הרץ: `*party-mode` לצ'אט קבוצתי עם כל האג'נטים

3. **Analyst (Mary)** - מחקר וניתוח דרישות
   - טען: `@analyst hi`
   - הרץ: `*product-brief` ליצירת Product Brief
   - הרץ: `*research` למחקר שוק או תחרות

**זרימת עבודה מלאה לספרינט:**

1. **תחילת ספרינט:**
   - הרץ `sprint-planning` לעדכון sprint-status.yaml
   - סקור Stories שצריכים הכנה
   - הרץ `create-story` לכל Story חדש שצריך

2. **במהלך ספרינט:**
   - בדוק sprint-status.yaml מדי יום
   - הרץ `story-context` ל-Stories שמוכנים לפיתוח
   - הרץ `correct-course` אם יש שינוי משמעותי

3. **סוף ספרינט:**
   - וודא שכל ה-Stories סומנו כ-Done
   - הרץ `retrospective` לרטרו מלאה עם הצוות
   - סקור Action Items מהרטרו
   - עדכן backlog לפי הלקחים

**טיפים מתקדמים:**
- הרץ `sprint-planning` כל בוקר לקבלת תמונת מצב עדכנית
- השתמש ב-`epic-tech-context` לאפיקים גדולים ומורכבים
- הרץ `retrospective` גם אם האפיק עדיין לא הסתיים - זה עדיין ייתן insights

### 🏃 Bob - הסקראם מאסטר הטכני שלך

**מה הוא עושה:**

Bob הוא ה-Scrum Master הטכני המומחה בהכנת Stories ותיאום פיתוח. הוא אחראי על:
- יצירת ועדכון sprint-status.yaml - קובץ המעקב המרכזי
- הכנת Story Drafts מפורטים מתוך Epics
- בניית Story Context XML עם כל המידע למפתחים
- הנחיית רטרוספקטיבות צוותיות אינטראקטיביות
- ניהול תיקון מסלול כשיש שינויים או בעיות

Bob שומר על גבול ברור: הוא מכין Stories, לא מממש אותם. הוא מבטיח שהמפתחים מקבלים מפרטים מושלמים.

**מתי להשתמש:**
- בתחילת כל ספרינט - `sprint-planning`
- כשצריך Story חדש - `create-story`
- לפני שמפתח מתחיל לעבוד - `story-context`
- בסיום Epic - `retrospective`
- כשיש בעיה בספרינט - `correct-course`

**דוגמה:**

אתה רוצה להכין את הספרינט הבא:
```
1. טען Bob: @sm hi
2. הרץ: *sprint-planning
3. Bob סורק את כל ה-Epics, מזהה Stories, ומעדכן sprint-status.yaml
4. בדוק את הקובץ - תראה את כל ה-Stories לפי Epic, סטטוס, ומי עובד על מה
5. בחר Story שצריך הכנה והרץ: *create-story
6. Bob יוצר Story Draft מפורט עם Acceptance Criteria ומשימות
7. הרץ: *story-context להכנת ה-XML למפתח
8. Bob אוסף מידע מ-PRD, Architecture, קוד קיים ויוצר Story Context שלם
```

**טיפ מקצועי:** Bob עובד בצורה לא-אינטראקטיבית ב-`create-story` - הוא משתמש ב-Architecture, PRD, ו-Epics ליצירת Draft מלא ללא שאלות. זה חוסך זמן עצום!

### 📋 John - מנהל המוצר האנליטי

**מה הוא עושה:**

John הוא Product Manager עם 8+ שנות ניסיון. הוא מומחה במחקר שוק, ניתוח תחרות, ותרגום דרישות עסקיות לתוכניות פיתוח. John אחראי על:
- יצירת PRD (Product Requirements Document) מקיפים
- פירוק PRD ל-Epics ו-Stories מובנים
- החלטות על prioritization ו-trade-offs
- ולידציה של PRD ו-Epics לפני שעוברים לפיתוח

John עובד עם גישה חוקרנית - הוא שואל שאלות מעמיקות כדי להבין את ה"למה" מאחורי כל דרישה.

**מתי להשתמש:**
- בתחילת פרויקט חדש - `create-prd`
- אחרי שה-PRD מוכן - `create-epics-and-stories`
- לולידציה של PRD - `validate-prd`
- כשצריך לבדוק סטטוס - `workflow-status`

**דוגמה:**

פרויקט חדש מתחיל:
```
1. טען John: @pm hi
2. הרץ: *workflow-status
3. John בודק אם יש Product Brief (אם לא, הוא ממליץ להתחיל שם)
4. הרץ: *create-prd
5. John מנתח את ה-Brief, שואל שאלות, ויוצר PRD מפורט
6. אחרי שה-PRD מוכן, הרץ: *create-epics-and-stories
7. John פורק את ה-PRD ל-Epics ברורים, כל Epic עם Stories מפורטים
8. עכשיו אתה (SM) יכול להשתמש ב-Bob ליצירת Story Drafts מהאפיקים האלה
```

**עבודה משותפת עם Bob:**
- John יוצר את ה-Epics → Bob יוצר Stories מתוכם
- John מחליט מה לבנות → Bob מחליט איך להכין את זה למפתחים
- John מנהל את ה-Product Vision → Bob מנהל את ה-Sprint Execution

### 1. workflow-status - המצפן שלך

**מטרה:** לענות על השאלה "מה אני צריך לעשות עכשיו?"

**איך זה עובד:**

הוורקפלואו בודק את מצב הפרויקט על ידי קריאת קובץ סטטוס YAML ומסמכי הפרויקט. הוא מזהה באיזה שלב אתה ומה הצעד הבא ההגיוני.

**כניסה:**
- מסמכי פרויקט קיימים (PRD, Architecture, Epics, Stories)
- קובץ סטטוס (אם קיים)

**יציאה:**
- המלצה ברורה: "הרץ workflow X" או "צריך לעשות Y"
- סיכום מה כבר נעשה ומה חסר

**שימוש:**
```
@sm hi
*workflow-status
```

Bob יבדוק ויגיד משהו כמו:
- "נראה שיש לך PRD ו-Architecture. מומלץ להריץ sprint-planning."
- "יש 5 Stories במצב Draft. הרץ story-context להכנתם."
- "Epic 2 הסתיים. זמן ל-retrospective!"

**טיפ:** הרץ את זה בתחילת כל יום או כשלא בטוח מה לעשות הלאה.

### 2. sprint-planning - הלב של ניהול הספרינטים

**מטרה:** לייצר ולעדכן את sprint-status.yaml - קובץ המעקב המרכזי על כל ה-Stories וה-Epics.

**איך זה עובד:**

הוורקפלואו סורק את כל קבצי ה-Epics, מזהה את כל ה-Stories שנוצרו (לפי Epic), ובונה/מעדכן קובץ sprint-status.yaml עם:
- רשימת כל ה-Stories מקובצים לפי Epic
- סטטוס כל Story (Draft, Approved, In Progress, Code Review, Done)
- מי עובד על איזה Story
- תאריכי התחלה וסיום

**כניסה:**
- קבצי Epic (בתיקיית docs/epics או שווה ערך)
- קבצי Story (בתיקיית stories/)

**יציאה:**
- sprint-status.yaml מעודכן במיקום המוגדר ב-config

**שימוש:**
```
@sm hi
*sprint-planning
```

Bob יסרוק, יזהה, ויכתוב:
```yaml
epics:
  - epic_id: "epic-1"
    title: "User Authentication"
    stories:
      - story_id: "story-1.1"
        title: "Login Page"
        status: "Done"
        assigned_to: "Dev1"
        started: "2024-01-15"
        completed: "2024-01-20"
      - story_id: "story-1.2"
        title: "Password Reset"
        status: "In Progress"
        assigned_to: "Dev2"
        started: "2024-01-18"
```

**מתי להריץ:**
- תחילת ספרינט חדש
- כל בוקר (לקבלת עדכון סטטוס)
- אחרי יצירת Stories חדשים
- כשרוצים לראות תמונת מצב כללית

**טיפ:** הרץ את זה מדי יום. זה לוקח 10 שניות ונותן לך visibility מלא.

### 3. create-story - יצירת Story מפורט

**מטרה:** ליצור Story Draft חדש מתוך Epic, מוכן לאישור ולפיתוח.

**איך זה עובד:**

הוורקפלואו משתמש בתבנית סטנדרטית ובמידע מה-Epic, PRD, ו-Architecture כדי לייצר Story Draft מלא עם:
- תיאור Story
- Acceptance Criteria מפורטים
- משימות (Tasks) לביצוע
- Definition of Done
- הערות טכניות

Bob עובד **לא-אינטראקטיבית** - הוא לא שואל שאלות, הוא יוצר Draft מלא אוטומטית.

**כניסה:**
- Epic שממנו ליצור את ה-Story
- PRD
- Architecture
- Tech Spec (אם קיים)

**יציאה:**
- קובץ Story Draft במבנה סטנדרטי (.md)
- הקובץ נשמר ב-stories/

**שימוש:**
```
@sm hi
*create-story
```

Bob שואל: "איזה Epic?" (או זיהוי אוטומטי)
Bob יוצר Story Draft:
```markdown
# Story 1.3: Two-Factor Authentication

## Description
User can enable 2FA...

## Acceptance Criteria
- AC-001: User can enable 2FA in settings
- AC-002: 2FA works with authenticator app
- AC-003: Backup codes provided

## Tasks
- Task-001: Create 2FA settings UI
- Task-002: Integrate authenticator library
- Task-003: Generate backup codes
...
```

**מתי להריץ:**
- אחרי שיצרת Epics ורוצה להתחיל ספרינט
- כשצריך Story חדש נוסף
- בתחילת כל איטרציה

### 4. story-context - הכנת המפתחים

**מטרה:** לאסוף את כל המידע הרלוונטי ל-Story אחד ולרכז אותו ב-Story Context XML אחד שהמפתח יטען.

**איך זה עובד:**

הוורקפלואו קורא את ה-Story Draft ואוסף מידע מ:
- PRD - הדרישות העסקיות
- Architecture - התכנון הטכני
- Epic-Tech-Context (אם קיים) - מפרט טכני ספציפי לאפיק
- קוד קיים רלוונטי
- תלויות APIs
- שיקולי אבטחה

הכל מרוכז ב-XML מובנה שהמפתח יכול לטעון ולעבוד עם Agent DEV.

**כניסה:**
- Story Draft (status = Draft או Approved)
- כל מסמכי הפרויקט

**יציאה:**
- Story Context XML (בתיקיית stories/)
- עדכון סטטוס Story ל-"Ready for Dev"

**שימוש:**
```
@sm hi
*story-context
```

Bob שואל: "איזה Story?"
Bob אוסף מידע, יוצר XML:
```xml
<story-context>
  <overview>...</overview>
  <acceptance-criteria>...</acceptance-criteria>
  <tasks>...</tasks>
  <technical-context>...</technical-context>
  <existing-code>...</existing-code>
  <apis>...</apis>
  <security>...</security>
</story-context>
```

**מתי להריץ:**
- אחרי ש-Story Draft אושר
- לפני שמפתח מתחיל לעבוד
- כשרוצים שהמפתח יקבל את כל הקונטקסט

**טיפ:** Story Context הוא המפתח לעבודת פיתוח יעילה. אל תדלג על זה!

### 5. retrospective - למידה ושיפור

**מטרה:** להריץ רטרוספקטיבה מקצועית ואינטראקטיבית אחרי Epic, עם כל הצוות.

**איך זה עובד:**

הוורקפלואו מריץ ישיבת רטרוספקטיבה מובנית בת 12 שלבים:
1. גילוי איזה Epic הושלם
2. ניתוח עמוק של כל ה-Stories (Dev Notes, Code Reviews, Lessons Learned)
3. טעינת רטרו קודמת (מעקב אחר action items)
4. תצוגה מקדימה של Epic הבא
5-6. **ישיבת רטרו אינטראקטיבית עם כל האג'נטים** (Party Mode!)
7. דיון על מוכנות לאפיק הבא
8. סינתזת action items
9. בדיקת מוכנות קריטית
10-12. סגירה, שמירה, עדכון

**כניסה:**
- Epic שהושלם (כל Stories = Done)
- sprint-status.yaml
- כל קבצי ה-Stories של האפיק

**יציאה:**
- מסמך רטרוספקטיבה מפורט (epic-X-retro-{date}.md)
- עדכון sprint-status.yaml

**שימוש:**
```
@sm hi
*epic-retrospective
```

**זרימת הרטרו:**
```
Bob: "Epic 1 הושלם. בואו נסכם."
Bob מציג מטריקות: 5 Stories, 3 שבועות, 2 אתגרים גדולים.

Bob: "מה היה טוב?"
Alice (PO): "זרימת Authentication עלתה על הציפיות!"
Charlie (Dev): "אסטרטגיית Caching חסכה 60% API calls!"
{user_name}: [אתה משתתף בדיון]

Bob: "מה היה קשה?"
Elena (Junior Dev): "נתקעתי עם database migrations..."
Charlie: "הdocs שלי היו ברורים!"
Bob: "{user_name}, מה דעתך?"
[אתה מסייע לפתור את הקונפליקט]

Bob: "בואו נגדיר action items..."
1. הוסף validation session לפני migrations (Owner: Charlie)
2. רפקטור authentication middleware (Priority: HIGH)
3. אם requirements משתנות, לעצור ולעדכן את כולם
```

**מתי להריץ:**
- אחרי Epic שלם
- לפני Epic חדש
- אחרי milestone משמעותי

### תרחיש 3: מעקב אחר Velocity הצוות

**בעיה:** רוצה להבין מה ה-Velocity של הצוות כדי לתכנן טוב יותר את הספרינטים הבאים.

**פתרון:**

1. **איסוף נתונים (5 דקות):**
```
@sm hi
*sprint-planning
```
Bob מעדכן sprint-status.yaml. פתח את הקובץ ותראה:
- Epic 1: 5 Stories, 3 שבועות, הושלם
- Epic 2: 8 Stories, במהלך, 4 Stories הושלמו ב-2 שבועות

2. **חישוב Velocity (ידני - 10 דקות):**
ספור Story Points (אם מוגדרים) או ספור Stories:
- Epic 1: 5 Stories / 3 שבועות = 1.67 Stories לשבוע
- Epic 2 (עד כה): 4 Stories / 2 שבועות = 2 Stories לשבוע

**Velocity ממוצע: ~1.8 Stories לשבוע**

3. **ניתוח מגמות (עם Mary - 15 דקות):**
```
@analyst hi
```
שאל את Mary לעזור בניתוח:
- למה ה-Velocity עלה באפיק 2? (יותר ניסיון, Story Context טוב יותר?)
- האם יש בעיות חוזרות? (בדוק Dev Notes בקבצי Story)
- מה ה-Velocity הצפוי לאפיק 3? (בהנחה של שיפור מתמשך)

Mary: "רואה ש-Stories עם Story Context הושלמו 40% מהר יותר. המשך עם זה!"

4. **תכנון לספרינט הבא (5 דקות):**
בהינתן Velocity = 1.8 Stories/שבוע:
- ספרינט של שבועיים → 3-4 Stories
- ספרינט של 3 שבועות → 5-6 Stories

עדכן את Backlog בהתאם - אל תעמיס יותר מהיכולת.

5. **תקשורת:**
שתף את הצוות:
- "ה-Velocity שלנו 1.8 Stories/שבוע"
- "לספרינט הבא נקח 4 Stories בעדיפות גבוהה"
- "נמשיך עם Story Context - זה משפר מהירות!"

**תוצאה:** תכנון ריאליסטי יותר, פחות over-commitment, צוות מרוצה יותר.

```
Charlie: "יש חוב טכני מאפיק 2 שצריך לטפל בו."
Dana: "צריך test infrastructure טוב יותר."

{user_name}: [אתה מחליט] "בוא נעשה prep sprint של שבוע לפני אפיק 3."
```

**שלב 6 - Action Items:**
```
Bob: "בואו נקבע action items קונקרטיים:"

**Process Improvements:**
1. הוסף edge cases checklist לכל Story Context
   Owner: Bob
   Deadline: לפני אפיק 3

**Technical Debt:**
1. רפקטור error handling framework
   Owner: Charlie
   Priority: HIGH, Estimated: 12 שעות

2. שפר תיעוד API
   Owner: Charlie
   Priority: MEDIUM, Estimated: 6 שעות

**Team Agreements:**
- כל Story צריך לכלול edge cases checklist
- תיעוד API הוא חובה, לא אופציה
```

3. **סיכום וסגירה (5 דקות):**
Bob שומר את מסמך הרטרו ב-`docs/retrospectives/epic-2-retro-{date}.md`
Bob מעדכן sprint-status.yaml: retrospective = "done"

**תוצאה:** רטרו מקצועית עם 3 action items קונקרטיים, accountability על רטרו קודמת, והחלטה על prep sprint. הצוות יודע מה לשפר ואיך להתכונן לאפיק הבא.

**Jira:** שמור Stories ב-Git, סנכרן סטטוסים בין sprint-status.yaml ל-Jira  
**Azure DevOps:** BMAD מנהל תוכן, הכלי מנהל tracking  
**GitHub Projects:** אותו עיקרון - שילוב מלא

### התאמה לצוות

**צוות קטן (2-5):** דלג על epic-tech-context, רטרו כל 2-3 אפיקים  
**צוות גדול (10+):** epic-tech-context לכל אפיק, רטרו אחרי כל אפיק

## ❓ שאלות נפוצות

**ש: BMAD מחליף Jira?**
ת: לא. BMAD משלים - מנהל תוכן ואיכות. Jira/Azure לtracking ודיווח.

**ש: כמה זמן לוקח להכין Story עם Context?**
ת: 5-10 דקות. חוסך שעות של הבהרות ושינויים.

**ש: מפתחים לא רוצים להשתמש ב-Story Context?**
ת: הראה חיסכון בזמן - מקצר פיתוח ב-30-40%.

**ש: צריך לעדכן sprint-status.yaml ידנית?**
ת: לא. הרץ `sprint-planning` והוא מעדכן אוטומטית.

**ש: ההבדל בין create-story ל-story-context?**
ת: create-story = Draft עם דרישות. story-context = מידע טכני למפתח.

**ש: כמה פעמים להריץ sprint-planning?**
ת: כל בוקר (10 שניות) לתמונת מצב עדכנית.

**ש: רטרו עם AI באמת עובד?**
ת: כן! האג'נטים מייצגים roles ומעלים נקודות שאתה עלול לפספס.

**ש: Story Context חסר מידע?**
ת: בדוק PRD ו-Architecture מלאים. Context שואב מהם.

**ש: retrospective באמצע Epic?**
ת: כן, יתריע שלא הסתיים. טוב לסקירת ביניים.

**ש: איך מטפלים ב-Technical Debt?**
ת: הרטרו מזהה debt. תעדוף בbacklog או הוסף debt-fix stories.

**ש: BMAD תומך Kanban?**
ת: כן. sprint-status.yaml עובד גם ל-Kanban flow.

**ש: הצוות לא דובר עברית?**
ת: שנה `communication_language` ל-English בconfig.

**ש: זמן ללמוד BMAD?**
ת: 3-4 שעות בסיס, שבוע לעבודה שוטפת, שבועיים לשליטה מלאה.
