## 🎯 למי מדריך זה מיועד

### תפקידים רלוונטיים
- **בודקי תוכנה (QA Engineers)** - בודקים ידניים ואוטומטיים המעוניינים לשפר תהליכי בדיקה
- **מהנדסי אוטומציה (Test Automation Engineers)** - מפתחי בדיקות אוטומטיות מחפשי יעילות
- **ארכיטקטי בדיקות (Test Architects)** - מתכננים אסטרטגיות בדיקה ברמת ארגון

### רקע נדרש
- הבנה בסיסית של מחזור חיי בדיקות (Test Lifecycle)
- היכרות עם מושגי Unit/Integration/E2E testing (רצוי)
- ניסיון עם Playwright, Cypress, Jest או כלי בדיקה דומה (לא חובה)

### מה תשיגו עם BMAD
- ✅ **תשתית בדיקות מקצועית** מ-zero תוך שעה
- ✅ **ATDD (Acceptance Test-Driven Development)** - בדיקות קודם, פיתוח אחר כך
- ✅ **אוטומציה מלאה** - CI/CD עם שערי איכות מבוססי נתונים
- ✅ **כיסוי בדיקות אופטימלי** - בדיקת סיכונים והקצאת משאבים חכמה

   - תקבלו: config files, מבנה תיקיות, Fixtures, Factories, בדיקות לדוגמה

2. סקרו את המבנה שנוצר:
   ```
   tests/
   ├── e2e/              # בדיקות E2E
   ├── api/              # בדיקות API
   ├── support/
   │   ├── fixtures/     # נתונים עם ניקוי אוטומטי
   │   └── factories/    # יצרני נתונים (faker)
   ```

3. הריצו את הבדיקות לדוגמה:
   ```bash
   npm run test:e2e
   ```

**אם יש פרויקט קיים בלי בדיקות:**
1. קראו [שילוב במערכות קיימות](../08-שילוב-במערכות-קיימות.md) - 30 דקות

### 1. testarch-framework - הקמת תשתית בדיקות

**מה זה עושה:**
מקים תשתית בדיקות מוכנה לייצור עם Playwright או Cypress - כולל קונפיגורציה, מבנה תיקיות, Fixtures, Factories, ובדיקות לדוגמה. פעם אחת לכל פרויקט.

**קלט:** package.json קיים (פרויקט ללא תשתית בדיקות)
**פלט:** תשתית מלאה - playwright.config.ts, מבנה tests/, fixtures, factories

**איך להריץ:**
```
@tea *framework
```

**דוגמה:**
פרויקט Next.js חדש ← Murat מתקין Playwright ← תוך 30 דקות יש תשתית מלאה עם בדיקות לדוגמה

### 2. testarch-test-design - תכנון אסטרטגיית בדיקות

**מה זה עושה:**
מתכנן אסטרטגיית בדיקות מקיפה עם הערכת סיכונים, סיווג עדיפויות (P0-P3), ובחירת רמות בדיקה (E2E/API/Component/Unit). מפה לכל דרישה: איזו בדיקה, באיזו עדיפות, וכמה זמן יקח.

**קלט:** Story עם acceptance criteria
**פלט:** מטריצת סיכונים, מטריצת כיסוי, קריטריוני quality gate, הערכת משאבים

**איך להריץ:**
```
@tea *testarch-test-design
```

**דוגמה:**
Story "Forgot Password" ← Murat מזהה 3 סיכונים (1 קריטי) ← מתכנן 7 בדיקות ← אומר שיקח 9 שעות

### 3. testarch-atdd - בדיקות לפני מימוש (TDD)

**מה זה עושה:**
יוצר בדיקות כושלות (RED phase) לפני מימוש התכונה. Murat כותב בדיקות שמכסות את כל ה-acceptance criteria, יוצר Fixtures ו-Factories, ומספק checklist למפתחים. הבדיקות ייכשלו עד שהמפתחים מממשים את התכונה.

**קלט:** Story מאושר + acceptance criteria + תשתית בדיקות
**פלט:** קבצי בדיקה failing, Fixtures, Factories, רשימת data-testid, Implementation Checklist

**איך להריץ:**
```
@tea *testarch-atdd
```

**דוגמה:**
Story "User Login" ← Murat יוצר 3 בדיקות failing ← מפתחים מממשים ← בדיקות עוברות (GREEN) ← מרפקטורים (REFACTOR)

---

### 4. testarch-automate - הרחבת כיסוי בדיקות

**מה זה עושה:**
מרחיב כיסוי בדיקות אחרי מימוש או מנתח codebase קיים ליצירת suite מקיף. תומך הן ב-BMad Mode (עם Story) והן ב-Standalone Mode (ניתוח קוד ישיר לbrownfield).

**קלט:** Story מומש OR קוד מקור קיים + תשתית בדיקות
**פלט:** בדיקות חדשות (E2E/API/Component), Factories נוספים, מסמך Automation Summary

**איך להריץ:**
```
@tea *testarch-automate
```

**דוגמה (Brownfield):**
אפליקציה קיימת עם 30% כיסוי ← Murat סורק קוד ← מזהה Shopping Cart ב-15% כיסוי ← יוצר 8 בדיקות ← כיסוי עולה ל-75%

---

### 5. testarch-ci - הקמת צינור בדיקות אוטומטי

**מה זה עושה:**
יוצר pipeline CI/CD מוכן לייצור עם הרצת בדיקות אוטומטית, burn-in loops, artifact collection, ודיווח. תומך ב-GitHub Actions, GitLab CI, Circle CI, Jenkins.

**קלט:** repository Git + תשתית בדיקות
**פלט:** קובץ CI (.github/workflows/test.yml או .gitlab-ci.yml)

**איך להריץ:**
```
@tea *testarch-ci
```

**דוגמה:**
פרויקט GitHub ← Murat יוצר .github/workflows/test.yml ← בדיקות רצות אוטומטית בכל PR ← artifacts נשמרים בכישלון

---

### 6. testarch-test-review - ביקורת איכות בדיקות

**מה זה עושה:**
סוקר איכות הבדיקות מול knowledge base של best practices. מזהה anti-patterns (hard waits, brittle selectors, hardcoded data), מציע שיפורים, ומוודא עמידה בתקנים.

**קלט:** קובץ/תיקיית בדיקות

## ❓ שאלות נפוצות

### 1. מה ההבדל בין testarch-atdd ל-testarch-automate?

**ת:** `atdd` = בדיקות **לפני** מימוש (RED phase, TDD). `automate` = בדיקות **אחרי** מימוש (הרחבת כיסוי לקוד קיים).

---

### 2. האם BMAD יכול לעבוד עם framework בדיקות קיים שלי?

**ת:** כן! אם יש לכם Playwright/Cypress, BMAD ישתמש בהם. אם אין, `testarch-framework` יתקין עבורכם. תומך גם ב-Jest, Vitest לunit tests.

---

### 3. כמה זמן לוקח להקים תשתית בדיקות מאפס?

**ת:** עם `testarch-framework` - **30-45 דקות** ויש לכם תשתית מלאה. ידנית זה היה לוקח 3-5 ימים.

---

### 4. מה זה ATDD ולמה זה חשוב?

**ת:** ATDD = Acceptance Test-Driven Development. כותבים בדיקות **לפני** הקוד. היתרונות: (1) בדיקות מגדירות התנהגות ברורה, (2) bugs מתגלים מוקדם, (3) כיסוי 100% מלכתחילה.

---

### 5. מה זה Quality Gate ואיך זה עובד?

**ת:** Quality Gate = החלטה אוטומטית אם אפשר לשחרר. Murat בודק: (1) P0 tests עוברות 100%, (2) P1 tests ≥95%, (3) High-risk items מטופלים. תוצאה: PASS / CONCERNS / FAIL / WAIVED.

---

### 6. איך BMAD מחליט בין Playwright ל-Cypress?

**ת:** Murat בוחר לפי גודל הפרויקט: פרויקט גדול (100+ קבצים) → Playwright (מקביליות, debugging טוב יותר). פרויקט קטן/בינוני → Cypress (DX טוב יותר). אפשר לדרוס ידנית.

---

### 7. מה זה Fixtures ו-Factories ולמה צריך אותם?

**ת:** **Factory** = מייצר נתוני בדיקה אקראיים (faker). **Fixture** = מנקה נתונים אוטומטית אחרי בדיקה. חשוב כי: (1) נתונים אקראיים מונעים התנגשויות, (2) ניקוי אוטומטי מונע זיהום בין בדיקות.

---

### 8. בדיקות נכשלות אקראית (Flaky Tests) - מה עושים?

**ת:** (1) הריצו `testarch-test-review` - Murat יזהה anti-patterns (hard waits, race conditions). (2) השתמשו ב-network-first pattern (יירוט לפני ניווט). (3) הריצו burn-in loop ב-CI (×3-5 runs) לזיהוי flakiness.

---

### 9. איך אני בודק כיסוי בדיקות?

**ת:** (1) הריצו `testarch-trace` - מקבלים traceability matrix שמקשרת AC לבדיקות. (2) הריצו `npm run test:coverage` (אם הגדרתם). (3) CI dashboard יראה כיסוי בכל PR.

---

### 10. מה זה NFR ואיך בודקים את זה?

**ת:** NFR = Non-Functional Requirements (ביצועים, אבטחה, אמינות, תחזוקתיות). הריצו `testarch-nfr` - Murat אוסף metrics, logs, security scans ומעריך מול targets. תוצאה: PASS/CONCERNS/FAIL לכל קטגוריה.

---

### 11. אפשר לכתוב בדיקות ב-TypeScript?

**ת:** כן! `testarch-framework` יתקין הכל עם TypeScript אם הפרויקט שלכם משתמש בו. ברירת מחדל: TypeScript אם יש `tsconfig.json`.

---

### 12. איך מתאמים בין בודקים למפתחים?

**ת:** תהליך: (1) אתם (QA) מריצים `testarch-atdd` - יוצרים בדיקות failing. (2) מפתחים מריצים `@dev dev-story` - מממשים עד שהבדיקות עוברות. (3) אתם מריצים `testarch-trace` - מוודאים כיסוי.

---

### 13. מה עושים אם בדיקות לוקחות יותר מדי זמן?

**ת:** (1) הריצו בדיקות במקביל (sharding ב-Playwright). (2) הריצו רק P0 בכל commit, P1 ב-PR, P2 nightly. (3) העבירו בדיקות E2E איטיות ל-API (מהיר יותר). (4) הריצו `testarch-test-review` - Murat יזהה בדיקות לא יעילות.

---

### 14. אפשר להשתמש ב-BMAD לבדיקות mobile?

**ת:** כרגע BMAD ממוקד ב-web (Playwright/Cypress). לmobile: אפשר להשתמש ב-API testing (`testarch-automate` על endpoints). לE2E mobile תצטרכו Appium בנפרד (BMAD לא תומך עדיין).

---

### 15. איך עוברים מ-manual testing ל-automation?

**ת:** (1) שבוע 1: הריצו `testarch-framework`. (2) שבוע 2-3: בחרו 5 תרחישים קריטיים שאתם עושים ידנית. (3) הריצו `testarch-automate` על כל אחד. (4) שבוע 4: הריצו `testarch-ci` - בדיקות אוטומטיות בכל PR. (5) בהדרגה: הוסיפו עוד בדיקות כל sprint.

| בעיה | תסמינים | פתרון מהיר |

| **Coverage נמוך** | <60% כיסוי | הריצו `testarch-automate` על features חסרים. הריצו `testarch-trace` לזהות gaps. |
| **CI pipeline נכשל** | GitHub Actions / GitLab CI error | בדקו `.nvmrc` (גרסת Node). בדקו environment variables (`.env.ci`). |

| **NFR Assessment חסר נתונים** | "No metrics available" | הריצו performance tests קודם. אספו logs מייצור. הריצו `npm audit` לsecurity. |
| **בדיקות לא נקיות (dirty data)** | בדיקה נכשלת בגלל נתונים מבדיקה קודמת | השתמשו ב-Fixtures עם auto-cleanup. בדקו שFactory.cleanup() נקרא. |
