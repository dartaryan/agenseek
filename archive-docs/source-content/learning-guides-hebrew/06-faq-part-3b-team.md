## 1. איך להגדיר BMAD לעבודה עם מספר מפתחים?

### ❓ שאלה

הצוות שלנו מורכב מכמה מפתחים. איך מגדירים את BMAD כך שכולם יוכלו לעבוד איתו בצורה אחידה?

**תשובה קצרה:** התקינו את BMAD פעם אחת בפרויקט, שתפו את תיקיית `bmad/` ב-Git, ותנו לכל מפתח להתאים את הגדרות ה-IDE האישיות שלו.

**תשובה מפורטת:**

ההגדרה הנכונה לצוות מורכבת משלושה שכבות: (1) **התקנת BMAD** - מספיק להריץ `npx @bmad/cli install` פעם אחת בפרויקט. זה יוצר את תיקיית `bmad/` עם כל האג'נטים, הוורקפלואים, והקונפיגורציה. (2) **שיתוף בגרסאות** - התיקייה `bmad/` נכנסת ל-Git כמו כל קוד אחר. כל המפתחים משתפים את אותם אג'נטים, וורקפלואים, ותצורה מרכזית. (3) **התאמות אישיות** - כל מפתח מתאים את ה-IDE שלו (Cursor, Claude Code, וכו') לטעון את האג'נטים. זה לא דורש סנכרון - כל אחד עובד עם ה-IDE שהוא אוהב.

החשוב להבין: BMAD היא מערכת שנמצאת בפרויקט, לא במחשב האישי. כולם עובדים מול אותם קבצים בדיוק כמו שעובדים מול אותו קוד. השינויים שאג'נט אחד יוצר (כמו PRD או Tech Spec) נשמרים בפרויקט ונגישים לכולם. אין צורך ב"סנכרון" או שרתים מרכזיים - Git מטפל בהכל.

**דוגמה:**

צוות של 5 מפתחים מתחיל פרויקט חדש. Sarah מריצה `npx @bmad/cli install`, בוחרת את המודול BMM (BMAD Method), ועושה commit של תיקיית `bmad/` ל-Git. המפתחים האחרים עושים `git pull`, מגדירים את ה-IDE שלהם (חלקם Cursor, חלקם Claude Code), ומיד כולם יכולים לטעון את אותם אג'נטים ולהריץ אותם וורקפלואים. כשדוד מריץ PRD עם John, השאר רואים את הקובץ ב-Git.

## 2. איך לשתף קונפיגורציה בין חברי צוות?

### ❓ שאלה

יש הגדרות שאני רוצה שיהיו זהות לכל הצוות - שפה, תיקיות פלט, סטנדרטים. איך משתפים את זה?

**תשובה קצרה:** ערכו את `bmad/core/config.yaml` והגדרות האג'נטים ב-`bmad/_cfg/agents/`, ושמרו ב-Git. כל הצוות יקבל את אותה קונפיגורציה.

**תשובה מפורטת:**

BMAD משתמשת בקבצי YAML לקונפיגורציה, וכולם נמצאים בתיקיית `bmad/` שמשותפת ב-Git. יש שתי רמות של הגדרות: (1) **הגדרות כלליות** ב-`bmad/core/config.yaml` - שפת תקשורת, שם המשתמש ברירת המחדל, תיקיית פלט למסמכים. אלה משפיעות על כל האג'נטים. (2) **הגדרות אג'נטים** ב-`bmad/_cfg/agents/*.yaml` - סגנון תקשורת של אג'נט מסוים, העדפות פלט, התאמות אישיות.

כשאתם עורכים את הקבצים האלה ועושים commit ל-Git, כל מפתח שעושה `git pull` מקבל את אותן הגדרות. אין צורך "לסנכרן" או להפעיל פקודה מיוחדת - זה פשוט חלק מהפרויקט. אבל שימו לב: לכל מפתח אפשר לעקוף הגדרות מקומיות אם הוא צריך - למשל, אם מישהו עובד בשפה אחרת או עם IDE שונה.

**דוגמה:**

הצוות של רותם מחליט שכל המסמכים צריכים להישמר ב-`docs/project-specs/` במקום `docs/`. רותם עורכת את `bmad/core/config.yaml` ומשנה `output_folder: '{project-root}/docs/project-specs'`. היא עושה commit ו-push. כל הצוות עושה pull ומעכשיו כל האג'נטים של כולם שומרים מסמכים באותה תיקייה.

## 3. איך לעשות onboarding לחבר צוות חדש ל-BMAD?

### ❓ שאלה

מפתח חדש הצטרף לצוות. איך להכין אותו לעבוד עם BMAD במהירות וביעילות?

**תשובה קצרה:** תנו לו להתקין את ה-IDE, לעשות clone לפרויקט (BMAD כבר שם), להגדיר את האג'נטים ב-IDE, ולהריץ demo workflow אחד. זה הכל.

**תשובה מפורטת:**

Onboarding ל-BMAD יותר פשוט ממה שחושבים כי BMAD כבר קיימת בפרויקט. התהליך המומלץ הוא: (1) **הכנה** - המפתח החדש מתקין IDE שתומך באג'נטים (Cursor, Claude Code, WindSurf וכו') ומקבל גישה לפרויקט ב-Git. (2) **Clone הפרויקט** - הוא עושה `git clone` ומקבל את תיקיית `bmad/` אוטומטית עם כל האג'נטים והוורקפלואים. (3) **הגדרת IDE** - הוא מגדיר את ה-IDE לטעון את האג'נטים מהתיקייה `.claude/commands/` או `.cursorrules/` (תלוי ב-IDE). (4) **Demo Workflow** - תנו לו להריץ workflow פשוט כמו `*list-workflows` עם BMad Master או סתם לדבר עם Amelia על הקוד. זה מראה שהכל עובד.

החלק החשוב הוא להסביר את הקונספט: האג'נטים הם "חברי צוות וירטואליים" עם תפקידים ספציפיים. כל אחד מתמחה בתחום - John ב-PM, Winston בארכיטקטורה, Amelia בקוד. תנו למפתח החדש לקרוא את המדריך המהיר (`00-התחלה-מהירה-לכולם.md`) ולהתנסות. תוך 30-60 דקות הוא יהיה מוכן לעבוד.

**דוגמה:**

טל הצטרפה לצוות. אבי (מפתח ותיק) אומר לה: "התקיני Cursor, עשי clone לפרויקט, ותגדירי את האג'נטים לפי המדריך שנמצא ב-`bmad/learning-guides-hebrew/00-התחלה-מהירה-לכולם.md`". טל קוראת את המדריך 15 דקות, מגדירה את Cursor, פותחת את BMad Master וכותבת "היי, מה אפשר לעשות פה?". Master מציג לה תפריט. אבי עובר איתה על PRD אחד שהצוות יצר ומראה לה איך Winston עשה את ה-Architecture. אחרי 45 דקות טל כבר מריצה workflow לבד.

## 4. איך לנהל BMAD עם version control?

### ❓ שאלה

מה צריך לעשות commit ל-Git? מה להתעלם? איך מטפלים בשינויים של BMAD?

**תשובה קצרה:** תעשו commit של כל תיקיית `bmad/` כולל קבצי הקונפיגורציה, האג'נטים, והוורקפלואים. התעלמו מקבצים זמניים או אישיים.

**תשובה מפורטת:**

BMAD מתוכננת לעבוד עם Git מהיום הראשון. כמעט הכל בתיקיית `bmad/` צריך להיות ב-version control: (1) **Commit למה?** - כל תיקיית `bmad/` כולל `bmad/core/`, `bmad/bmm/`, `bmad/_cfg/`, ומסמכים שנוצרו על ידי אג'נטים (PRD, Tech Spec, Architecture וכו'). אלה הם חלק מהפרויקט ממש כמו קוד. (2) **התעלמות ממה?** - קבצים זמניים כמו `*.log`, `*.tmp`, או קבצי IDE אישיים (`.idea/`, `.vscode/settings.json` אישי). BMAD מגיעה עם `.gitignore` מוכן.

כשאג'נט משנה קונפיגורציה או יוצר מסמך, זה שינוי שאתם עושים commit ל-Git כמו כל שינוי אחר. אם שני מפתחים עורכים את אותו קובץ YAML, Git יתפוס את זה כקונפליקט רגיל. תפתרו אותו כמו כל קונפליקט - ידנית או עם merge tool.

**דוגמה:**

יונתן עורך את `bmad/_cfg/agents/dev.yaml` כדי להוסיף העדפה שכל הקוד צריך להיות עם TypeScript. הוא עושה commit: `git commit -m "feat: configure dev agent for TypeScript preference"`. דנה עושה pull, רואה את השינוי, ועכשיו כשהיא עובדת עם Amelia - האג'נט יודע לכתוב TypeScript. אם דנה הייתה עורכת את אותו קובץ במקביל, Git היה מתריע על conflict והם היו פותרים את זה ביחד.

## 5. מהן שיטות עבודה מומלצות לצוות שעובד עם BMAD?

### ❓ שאלה

רוצים לעבוד עם BMAD בצורה מקצועית ויעילה. מה הן ה-best practices לצוות?

**תשובה קצרה:** קבעו מי מריץ איזה workflow, תעשו review למסמכים שאג'נטים יוצרים, שמרו על אותם סטנדרטים, ותקשרו שינויים בצוות.

**תשובה מפורטת:**

עבודה צוותית עם BMAD דורשת תיאום כמו כל כלי אחר. הנה העקרונות החשובים: (1) **חלוקת תפקידים** - קבעו מי אחראי על מה: למשל, ה-PM מריץ PRD workflows עם John, האדריכל מריץ Architecture עם Winston, המפתחים עובדים עם Amelia. זה מונע כפילויות ובלבול. (2) **Review למסמכים** - כשאג'נט יוצר PRD או Tech Spec, תעשו code review למסמך בדיוק כמו לקוד. האג'נטים טובים אבל לא מושלמים. (3) **סטנדרטים אחידים** - הגדירו בקבצי התצורה את הסטנדרטים של הצוות (TypeScript, React patterns וכו') כך שכל האג'נטים יפיקו קוד עקבי. (4) **תקשורת** - כשמישהו משנה קונפיגורציה או מריץ workflow משמעותי, תעדכנו את הצוות ב-Slack או כלי התקשורת שלכם.

נוסף על זה, תשתמשו בוורקפלואים המובנים במקום בקשות חופשיות - זה שומר על עקביות. אם הצוות תמיד משתמש ב-`workflow-prd` של John, תקבלו תמיד אותה מבנה PRD. זה עוזר לכולם להבין במהירות את המסמכים.

**דוגמה:**

הצוות של ליאור קובע: (1) PM (אורלי) אחראית על כל ה-PRD workflows. (2) Tech Lead (ליאור) אחראי על Architecture. (3) מפתחים משתמשים ב-Amelia לקוד ו-Murat לטסטים. (4) כל מסמך שנוצר עובר review במשך יום לפני שממשיכים הלאה. (5) שינויים בקונפיג דורשים הסכמה ב-Slack לפני commit. אחרי שבועיים הצוות עובד כמו שעון - אין בלבול, המסמכים איכותיים, והקוד עקבי.

## 6. איך לשמור על עקביות בין מפתחים שונים?

### ❓ שאלה

כל מפתח עובד עם האג'נטים בצורה קצת שונה. איך מוודאים שהפלט יהיה עקבי?

**תשובה קצרה:** הגדירו את העדפות הצוות בקבצי התצורה של האג'נטים, השתמשו בוורקפלואים המובנים, ותעשו code review גם למסמכים.

**תשובה מפורטת:**

עקביות מושגת דרך שלושה מנגנונים: (1) **תצורה משותפת** - ערכו את קבצי האג'נטים ב-`bmad/_cfg/agents/` והוסיפו הנחיות ברורות: "Always use TypeScript", "Follow Airbnb style guide", "Include unit tests for all functions". כל מפתח שמשתמש באותו אג'נט יקבל אותן הנחיות. (2) **וורקפלואים סטנדרטיים** - במקום שכל אחד יבקש מ-John "תכתוב PRD" בסגנון חופשי, תמיד השתמשו ב-`workflow-prd`. הוורקפלואו מנחה את האג'נט בתהליך קבוע שמייצר פלט עקבי. (3) **Review תהליך** - תעשו pull request reviews גם למסמכים שאג'נטים יצרו. אם דוד קיבל מ-Winston architecture שונה מהסגנון שהצוות רגיל, תשאלו למה ותתקנו.

בנוסף, חשוב שהצוות יעבוד באותו "שפה" - אם כולם קראו את המדריכים של BMAD, יש להם אותו מושג על מה כל אג'נט עושה ומתי להשתמש בו. זה מונע מצב שאחד משתמש ב-Amelia למשהו ש-Winston צריך לעשות.

**דוגמה:**

הצוות של שירה שם לב שכל מפתח מקבל סגנון קוד קצת שונה מ-Amelia. שירה פותחת את `bmad/_cfg/agents/dev.yaml` ומוסיפה סעיף `code_standards: "Use Prettier with config in .prettierrc, ESLint with Airbnb rules, always include JSDoc comments"`. היא עושה commit. מעכשיו כל מפתח שעובד עם Amelia מקבל קוד שעומד בדיוק באותם סטנדרטים. הצוות גם מחליט שכל Tech Spec עובר review של שני מפתחים לפני שממשיכים לקוד.

## 7. מה לעשות כשיש קונפליקטים בקבצי BMAD?

### ❓ שאלה

שני מפתחים ערכו את אותו קובץ YAML של אג'נט. Git מראה conflict. איך פותרים את זה?

**תשובה קצרה:** פתרו את זה כמו כל merge conflict - בדקו את שני הגרסאות, שלבו את השינויים הנחוצים, ועשו commit. אין כאן קסם.

**תשובה מפורטת:**

קונפליקטים בקבצי BMAD זה בדיוק כמו קונפליקטים בקוד - Git מזהה שני אנשים ערכו את אותו חלק. התהליך לפתרון: (1) **פתחו את הקובץ** - Git מסמן את הקונפליקט ב-`<<<<<<<`, `=======`, `>>>>>>>`. אתם רואים את גרסת ה-HEAD ואת גרסת הענף השני. (2) **הבינו את השינויים** - קראו מה כל אחד שינה. לפעמים זה שני שינויים עצמאיים (אחד הוסיף preference, השני הוסיף אחר) - אז תשמרו את שניהם. לפעמים זה שינויים מנוגדים - אז צריך להחליט איזה נכון. (3) **שלבו ידנית** - ערכו את הקובץ כך שיכלול את הגרסה הנכונה, מחקו את הסימנים של Git, ועשו commit.

אם השינויים גדולים או מורכבים, תשבו יחד עם המפתח השני ותפתרו ביחד. לפעמים גם שווה לקרוא את הקובץ המלא ולוודא שהוא עדיין הגיוני אחרי המיזוג. YAML רגיש למבנה אז תבדקו שהוא תקין (אפשר להריץ `npm run validate:schemas` אם יש לכם).

**דוגמה:**

גלעד ורחל ערכו את `bmad/_cfg/agents/dev.yaml`. גלעד הוסיף `prefer_tests: true`, רחל הוסיפה `code_style: functional`. כשרחל עושה pull, Git מתריע על conflict. היא פותחת את הקובץ, רואה שהשינויים לא מתנגשים (גלעד ערך שדה אחד, היא שדה אחר), משלבת את שניהם, ועושה commit. אם היו שניהם שינו את אותו שדה (למשל גלעד `code_style: functional`, רחל `code_style: OOP`), היא הייתה שואלת את גלעד "מה ההבדל ביניהם?" והם היו מחליטים ביחד.

**קישורים רלוונטיים:**
- [Git Merge Conflicts - Atlassian Guide](https://www.atlassian.com/git/tutorials/using-branches/merge-conflicts)
- [CONTRIBUTING.md](../../CONTRIBUTING.md#pull-request-process)

## 8. איך להתאים BMAD לסטנדרטים של הצוות שלנו?

### ❓ שאלה

לצוות שלנו יש סטנדרטים ספציפיים - מבנה קוד, naming conventions, תהליכי review. איך BMAD מתאימה לזה?

**תשובה קצרה:** ערכו את קבצי התצורה של האג'נטים והוסיפו את הסטנדרטים שלכם. האג'נטים יפעלו לפיהם.

**תשובה מפורטת:**

BMAD נועדה להיות גמישה ולהתאים לכל צוות. כך מתאימים אותה לסטנדרטים שלכם: (1) **סטנדרטי קוד** - פתחו את `bmad/_cfg/agents/dev.yaml` (זה Amelia) והוסיפו בשדה `instructions` או `preferences` את הסטנדרטים: "Always use named exports", "Prefix private methods with underscore", "React components must have PropTypes". Amelia תפעל לפי זה. (2) **מבנה מסמכים** - אפשר לערוך את הוורקפלואים ב-`bmad/bmm/workflows/` ולשנות את תבניות הפלט. למשל, אם אתם רוצים PRD עם סעיף נוסף "Compliance Requirements", תוסיפו אותו לתבנית. (3) **תהליכי עבודה** - אפשר להוסיף שלבים לוורקפלואים או ליצור וורקפלואים חדשים שמשקפים את התהליך שלכם.

בנוסף, אפשר להתאים שמות, שפה, וטון של האג'נטים. אם הצוות שלכם עובד בעברית, אפשר לשנות את `communication_language: Hebrew` ב-`config.yaml`. אם אתם רוצים שאג'נט יהיה יותר פורמלי או קז'ואלי, אפשר לשנות את ה-`communication_style` שלו.

**דוגמה:**

הצוות של נועה עובד עם Vue.js ויש להם סטנדרט: components בתיקייה נפרדת, Composition API בלבד, TypeScript חובה. נועה פותחת את `bmad/_cfg/agents/dev.yaml` ומוסיפה:
```yaml
code_preferences: |
  - Use Vue 3 Composition API exclusively
  - Components in /src/components/ with PascalCase naming
  - TypeScript required for all files
  - Use defineProps and defineEmits
```
עכשיו כשמישהו מהצוות עובד עם Amelia, האג'נט יוצר קוד Vue שמתאים בדיוק לסטנדרט. הצוות גם מוסיף בוורקפלואו `workflow-code-review` שלב שבודק Vue best practices.

## 9. איך לשלב BMAD עם תהליכי העבודה הקיימים שלנו?

### ❓ שאלה

הצוות שלנו כבר עובד עם Jira, GitHub Projects, Scrum. איך BMAD משתלבת עם התהליכים האלה?

**תשובה קצרה:** BMAD משלימה את התהליכים הקיימים - האג'נטים יוצרים את המסמכים והקוד, ואתם מנהלים את המשימות ב-Jira/GitHub כרגיל.

**תשובה מפורטת:**

BMAD היא מערכת תיעוד ועבודה עם AI, לא כלי ניהול פרויקטים. היא משתלבת עם התהליכים שלכם ככה: (1) **Jira/GitHub Issues** - המשימות ממשיכות להיות שם. כשאתם מתכננים Sprint, אתם יוצרים Issues ב-Jira. רק שעכשיו, במקום לכתוב את ה-PRD ידנית, אתם מריצים את John שעושה את זה בעזרת AI. התוצר (PRD.md) מקושר מה-Jira ticket. (2) **Scrum ceremonies** - Daily standups, Retros, Planning - הכל נשאר. האג'נטים עוזרים במה שהם טובים בו (יצירת מסמכים, כתיבת קוד, ארכיטקטורה), אבל הצוות עדיין מחליט מה לבנות ואיך לתעדף. (3) **Code review** - Pull requests ב-GitHub או Bitbucket נשארים כרגיל. רק שהקוד נכתב עם עזרת Amelia, והמסמכים שמסבירים אותו (Tech Spec) נוצרו עם Winston.

המפתח הוא להבין ש-BMAD לא מחליפה כלום - היא מוסיפה שכבת AI לתהליך. אתם ממשיכים לעבוד כרגיל, רק מהיר ויותר יעיל. המסמכים שאג'נטים יוצרים אפשר לצרף ל-Jira tickets, לשתף בקישורים ב-Confluence, לשמור ב-Google Docs - מה שאתם רוצים.

**דוגמה:**

הצוות של איתי עובד עם Jira ו-GitHub. בתחילת Sprint, ה-PM (דנה) מריצה את John ליצירת PRD לתכונה חדשה. היא שומרת את ה-PRD.md ומעלה אותו ל-Confluence, ואז יוצרת Epic ב-Jira עם קישור למסמך. האדריכל (איתי) מריץ את Winston ליצירת Tech Spec, ושומר ב-Git. המפתחים רואים את ה-PRD וה-Tech Spec, יוצרים Stories ב-Jira, ועובדים איתם עם Amelia. הקוד עובר code review ב-GitHub PR. השילוב שקוף לחלוטין.

## 10. איך להרחיב BMAD לצוותים גדולים?

### ❓ שאלה

הצוות שלנו כולל 20+ מפתחים ומספר צוותי פיתוח. איך משתמשים ב-BMAD בסקייל כזה?

**תשובה קצרה:** השתמשו בפרויקטים נפרדים או מונורפו עם תצורות משותפות, הקימו "BMAD champions" שמדריכים, ושמרו על סטנדרטים מרכזיים.

**תשובה מפורטת:**

סקייל ל-BMAD דומה לסקייל של כל תשתית פיתוח: (1) **מבנה פרויקטים** - צוותים גדולים לרוב עובדים במונורפו או במספר repositories. אם זה מונורפו, תיקיית `bmad/` אחת בשורש משרתת את כולם. אם זה repos נפרדים, כל repo יכול לקבל את ה-BMAD שלו עם תצורה משותפת (אפשר גם להשתמש ב-git submodules או packages משותפים). (2) **BMAD Champions** - מנו אנשים שמתמחים ב-BMAD ומדריכים אחרים. הם יוצרים סטנדרטים, מעדכנים תצורות, ומסייעים בעיות. זה כמו DevOps champions או Security champions. (3) **תצורות מרכזיות** - שמרו את קבצי התצורה של האג'נטים ב-repo מרכזי והפיצו אותם לכל הצוותים. כך כולם עובדים עם אותם סטנדרטים.

בנוסף, כדאי ליצור מדריכים פנימיים ספציפיים לארגון שלכם - איך להשתמש ב-BMAD בהקשר של המוצר שלכם, עם דוגמאות מהפרויקטים שלכם. תעשו workshops ו-onboarding sessions למפתחים חדשים. ככל שיותר אנשים יודעים איך להשתמש ב-BMAD, כך התועלת גדלה.

**דוגמה:**

חברה בת 50 מפתחים עם 5 צוותי מוצר מחליטה להשתמש ב-BMAD. הם יוצרים repo מרכזי `bmad-config` עם תצורות אג'נטים ווורקפלואים מותאמים. כל צוות מוצר מוסיף את זה כ-git submodule או מעתיק את התיקייה לפרויקט שלו. הם מגדירים 3 "BMAD Champions" (אחד PM, אחד architect, אחד dev) שעושים workshop חודשי ועוזרים לצוותים. הם גם יוצרים `#bmad-help` ב-Slack. תוך 3 חודשים, כל 50 המפתחים עובדים עם BMAD, חוסכים שעות בתיעוד, ומקבלים קוד יותר עקבי.
