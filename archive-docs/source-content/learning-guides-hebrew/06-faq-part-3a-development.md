## ❓ איך מטפלים בבעיות איכות קוד?

**תשובה קצרה:** BMAD מספקת וורקפלואים מיוחדים לסקירת קוד ושיפור איכות, עם Definition of Done מובנה ואג'נטים שמזהים בעיות לפני merge.

**תשובה מפורטת:**

כשעובדים עם BMAD, איכות הקוד מובטחת דרך מספר מנגנונים. ראשית, כל וורקפלואו פיתוחי (כמו story-dev) כולל צ'קליסט מובנה של Definition of Done שבודק שהקוד עומד בסטנדרטים: יש טסטים, יש תיעוד, ואין code smells. שנית, האג'נט Amelia יודעת לכתוב קוד נקי לפי Best Practices ומזהירה על בעיות פוטנציאליות. שלישית, יש וורקפלואו code-review שמנחה אתכם לבצע סקירה שיטתית.

אם אתם מגלים בעיות איכות בקוד קיים, אתם יכולים להשתמש בוורקפלואו refactor - הוא עוזר לזהות code smells, להציע שיפורים, ולבצע רפקטורינג בטוח עם כיסוי טסטים. האג'נטים גם מסוגלים לזהות דפוסי עיצוב שגויים ולהציע אלטרנטיבות מובנות יותר.

**דוגמה:**

אלון כתב פונקציה של 200 שורות שעושה יותר מדי דברים. הוא טוען את Amelia, מריץ את code-review, והיא מזהה שהפונקציה מפרה את Single Responsibility Principle. היא מציעה לפצל אותה ל-4 פונקציות קטנות, כותבת את הרפקטורינג, ודואגת שכל הטסטים הקיימים עדיין עוברים.

## ❓ איך מנהלים סטוריז בפרקטיקה?

**תשובה קצרה:** BMAD מספקת וורקפלואים ליצירה, תיעדוף, ומעקב אחר סטוריז, עם תבניות מוכנות וחיבור ישירות לכלי ניהול כמו Jira.

**תשובה מפורטת:**

ניהול סטוריז ב-BMAD הוא תהליך מובנה שמתחיל בוורקפלואו user-stories שיוצר סטוריז מפורטים מאפיקים. כל סטורי מכיל קריטריוני קבלה ברורים, הערכת מאמץ, ותלויות. אחר כך אתם משתמשים בוורקפלואו sprint-plan כדי לתעדף ולהקצות סטוריז לספרינט.

במהלך הספרינט, האג'נט Kate (Scrum Master) עוזרת לעקוב אחרי התקדמות, לזהות חסמים, ולעדכן סטטוסים. אם אתם משתמשים ב-Jira או כלי ניהול אחר, אתם יכולים לייצא את הסטוריז שנוצרו ישירות או להשתמש בהם כמקור לעדכון ידני.

**דוגמה:**

לצוות של שרה יש 20 סטוריז בתור ה-backlog לספרינט. הם מריצים sprint-plan עם Kate - היא עוזרת להם לסדר לפי עדיפות, לזהות שיש סטורי שתלוי באחר, ולהפריד סטורי גדול מדי לשניים. בסוף יש להם ספרינט מתוכנן עם 8 סטוריז מאוזנים.

## ❓ איך משלבים BMAD עם מערכת בדיקות?

**תשובה קצרה:** BMAD כוללת אג'נט מיוחד לבדיקות (Murat) עם וורקפלואים לכתיבת טסטים, אוטומציה, ותכנון test coverage, שמשתלבים עם כלי הטסטינג הקיימים שלכם.

**תשובה מפורטת:**

BMAD נבנתה עם Testing First Approach - האג'נט Murat The Lead TEA (Test & Evaluation Agent) מומחה בכל סוגי הבדיקות: unit tests, integration tests, E2E, וטסטי ביצועים. יש וורקפלואים מיוחדים כמו test-plan שיוצר אסטרטגיית בדיקות מקיפה, ו-test-automation שמייצר סוויטת טסטים אוטומטית.

כשאתם מפתחים סטורי חדש, הוורקפלואו story-dev כולל שלב של כתיבת טסטים - Murat כותב את הטסטים לפי הפרמקרוורק שאתם משתמשים בו (Jest, Pytest, JUnit וכו'). הטסטים נכתבים עם הקוד ולא אחריו, מה שמבטיח כיסוי טוב. אתם יכולים להריץ את הטסטים במערכת ה-CI שלכם כרגיל - הם קבצים רגילים.

לבעיות מורכבות יותר כמו API testing או performance testing, יש וורקפלואים מיוחדים שמנחים אתכם לבנות מערכת בדיקות שלמה עם mocks, fixtures, וסביבות test.

**דוגמה:**

צוות של רועי מפתח API חדש. הם מריצים test-plan עם Murat - הוא מציע אסטרטגיה: unit tests לכל endpoint, integration tests למסד נתונים, וכמה E2E tests למסלולים קריטיים. הוא כותב את הטסטים בPostman עם Newman לאוטומציה, והצוות מוסיף אותם ל-GitLab CI pipeline.

## ❓ איך מטפלים בהזיות של AI?

**תשובה קצרה:** הזיות (hallucinations) של AI נפוצות אבל ניתן למנוע אותן על ידי בדיקת פלט, עבודה עם וורקפלואים מובנים, וסקירה קפדנית של הקוד שנוצר.

**תשובה מפורטת:**

הזיות (מצב שבו ה-AI ממציא דברים שלא קיימים) הן בעיה מוכרת בעבודה עם AI. ב-BMAD יש מספר מנגנוני הגנה: ראשית, כל וורקפלואו כולל שלב של Validation שבודק שהפלט הגיוני. שנית, האג'נטים מאולפים לבקש הבהרות במקום להמציא - אם John לא יודע משהו, הוא ישאל אתכם במקום לנחש. שלישית, יש צ'קליסטים מובנים שדורשים ממכם לבדוק פלט לפני להמשיך.

עם זאת, תמיד צריך לבדוק את הקוד שנוצר - אל תסמכו 100% על ה-AI. הריצו את הקוד, בצעו טסטים, וודאו שהתוצאה תואמת את הציפיות. אם האג'נט ממליץ על ספרייה או API שלא מוכרים לכם, בדקו בתיעוד הרשמי שזה באמת קיים ופועל ככה.

אם אתם מזהים הזיה (למשל, פונקציה שלא קיימת או דפוס עיצוב בדוי), תקנו אותה מיד וציינו זאת באינטראקציה הבאה עם האג'נט - זה עוזר לו ללמוד ולהשתפר. אפשר גם לבקש מהאג'נט לצטט מקורות או לספק לינקים לתיעוד כדי לאמת מידע.

**דוגמה:**

מיכל מריצה architecture workflow עם Winston. הוא מציע להשתמש ב"React Quantum State Manager" - ספרייה שנשמעת מגניב אבל לא קיימת. מיכל חוקרת ולא מוצאת אותה בnpm. היא חוזרת לWinston ואומרת "הספרייה הזאת לא קיימת - אנא הצע חלופה אמיתית." הוא מתנצל ומציע Redux Toolkit במקום.

## ❓ מה עושים עם בעיות ביצועים?

**תשובה קצרה:** BMAD לא אמורה להשפיע על ביצועי האפליקציה שלכם - אם יש בעיות, בדקו את הקוד שנוצר, הריצו פרופיילינג, והשתמשו באג'נטים לאופטימיזציה.

**תשובה מפורטת:**

BMAD היא כלי פיתוח - היא לא רצה בתוך האפליקציה שלכם בזמן ריצה, אז היא לא יכולה להוריד ביצועים. אם יש בעיות ביצועים, זה קשור לקוד עצמו. האג'נטים כותבים קוד שאמור להיות יעיל, אבל לפעמים יש optimizations שדורשים ידע ספציפי על המערכת שלכם.

אם אתם מזהים בעיית ביצועים, הריצו profiling כדי למצוא את הצוואר בקבוק. אחר כך תעלו את התוצאות לאג'נט רלוונטי (Amelia למשל) ובקשו סיוע באופטימיזציה. האג'נטים יודעים דפוסי אופטימיזציה מתקדמים: caching, lazy loading, memoization, indexing במסד נתונים, ועוד.

**דוגמה:**

לאפליקציה של דן יש עמוד שנטען איטי. הוא משתמש ב-Chrome DevTools ומגלה ש-API call אחד לוקח 3 שניות. הוא טוען את Amelia, מראה לה את הקוד, והיא מזהה שיש N+1 query problem. היא מציעה לשנות ל-eager loading, ומוסיפה index למסד נתונים - הזמן יורד ל-200ms.

## ❓ איך מנפים באגים מורכבים עם BMAD?

**תשובה קצרה:** טענו את Amelia או Murat עם הקוד הרלוונטי, תארו את הבאג בפירוט, והם יעזרו לזהות את הסיבה ולהציע תיקון.

**תשובה מפורטת:**

ניפוי באגים עם BMAD הוא תהליך שיתופי. כשאתם נתקלים בבאג, התחילו מלאסוף מידע: מה הסימפטום, מתי זה קורה, מה היה צפוי לקרות. אחר כך טענו את Amelia (לבאגים בקוד) או Murat (לבאגים בזרימות מורכבות), שתפו את הקוד הרלוונטי, ותארו את הבעיה.

האג'נטים יודעים לנתח stack traces, לזהות דפוסי שגיאה נפוצים, ולהציע היפותזות. הם יכולים להציע לכם איך לבודד את הבעיה (איזה breakpoints לשים, איזה לוגים להוסיף), ואחר כך לעזור לכתוב את התיקון עם טסט שמוודא שהבאג לא יחזור.

**דוגמה:**

שי מקבל שגיאת race condition באפליקציית React - לפעמים הדאטה לא מתעדכן נכון. הוא טוען את Amelia, מראה לה את הקומפוננטים והוורקפלואו. היא מזהה שיש useEffect ללא dependencies array נכון, מה שגורם לעדכונים מרובים. היא מתקנת את הבעיה וכותבת טסט שמוודא שהעדכון קורה רק פעם אחת.

## ❓ איך מנהלים חוב טכני בעזרת BMAD?

**תשובה קצרה:** BMAD עוזרת לזהות חוב טכני דרך code reviews אוטומטיות, ומספקת וורקפלואים לתיעדוף ורפקטורינג מדורג.

**תשובה מפורטת:**

ניהול חוב טכני מתחיל בזיהוי: הריצו code review על חלקי הקוד הקיים, והאג'נטים יזהו code smells, duplication, בעיות ארכיטקטורה, ותיעוד חסר. אתם יכולים ליצור רשימה של פריטי חוב טכני ולתעדף אותם לפי השפעה וסיכון.

אחר כך השתמשו בוורקפלואו refactor לטפל בפריטים בצורה מבוקרת: תכננו את הרפקטורינג, כתבו טסטים שמגנים על הפונקציונליות, בצעו את השינוי בהדרגה, וודאו שהכל עובד. האג'נטים עוזרים לוודא שהרפקטורינג לא שובר דברים ושהקוד משתפר באמת.

**דוגמה:**

צוות של תמר זיהה שיש להם 5 מודולים עם קוד כמעט זהה (copy-paste). הם מריצים refactor עם Amelia - היא מציעה ליצור מודול משותף, לחלץ את הלוגיקה המשותפת, ולהשאיר רק את ההבדלים ספציפיים. הרפקטורינג מקטין את הקוד ב-40% ומקל על תחזוקה.

## ❓ איך משלבים BMAD בתהליך Code Review?

**תשובה קצרה:** השתמשו בוורקפלואו code-review לפני ה-PR, שמבצע בדיקה אוטומטית ומציע שיפורים, ואז בצעו סקירה אנושית לפני merge.

**תשובה מפורטת:**

Code review ב-BMAD היא תהליך דו-שלבי: שלב ראשון הוא סקירה אוטומטית - לפני שאתם פותחים Pull Request, הריצו את code-review עם Amelia או Murat. הם יבדקו את הקוד לפי צ'קליסט מקיף: convention, readability, security, performance, test coverage, ותיעוד. אתם מקבלים רשימת ממצאים עם המלצות לשיפור.

אחרי שתיקנתם את הבעיות שנמצאו, אתם פותחים PR לסקירה אנושית. סקירת האדם מתמקדת בהיגיון העסקי, בהחלטות עיצוב, ובהקשר - דברים שה-AI לא תמיד יכולה להעריך. כך אתם חוסכים זמן לסוקרים האנושיים ומבטיחים שהקוד באיכות גבוהה.

**דוגמה:**

יעל מסיימת לכתוב פיצ'ר חדש. לפני ה-PR, היא מריצה code-review עם Amelia - מתגלה שהיא שכחה להוסיף error handling במקום אחד, ושיש פונקציה שלא מתועדת. יעל מתקנת, ואז פותחת PR. הסוקר האנושי מתמקד בבדיקת הלוגיק העסקי והסקירה חלקה הרבה יותר.

## ❓ איך BMAD עובדת עם Version Control?

**תשובה קצרה:** BMAD עובדת מעל Git ולא מחליפה אותו - האג'נטים עוזרים לכתוב commit messages טובים, לנהל branches, ולתכנן merge strategies.

**תשובה מפורטת:**

BMAD לא מחליפה את Git - היא עובדת איתו. כשאתם עובדים עם האג'נטים בעורכי קוד כמו Cursor, הם יכולים לבצע פעולות Git בשבילכם (לפתוח branch, לעשות commit, לפתוח PR) או פשוט לעזור לכם להחליט על האסטרטגיה.

למשל, האג'נטים יכולים לעזור לכם לכתוב commit messages ברורים לפי Conventional Commits, לתכנן branching strategy (Git Flow, Trunk-Based, וכו'), ולהחליט איך למזג שינויים מורכבים. אם יש merge conflict, אתם יכולים לבקש עזרה מהאג'נט להבין מה השתנה ואיך לפתור את הקונפליקט.

**דוגמה:**

לצוות של אבי יש שתי תכונות מקבילות שמשנות את אותו קובץ. לפני המרג', אבי טוען את Amelia ומראה לה את שני הברנצ'ים. היא מנתחת את השינויים ומציעה merge strategy: קודם למזג branch A, להריץ טסטים, אחר כך למזג branch B עם תיקון ידני בשורות 45-52. אבי עוקב אחרי ההמלצה וה-merge עובר חלק.

**קישורים רלוונטיים:**

## ❓ איך משלבים BMAD עם CI/CD?

**תשובה קצרה:** הקוד שנוצר עם BMAD הוא קוד רגיל שעובד עם כל מערכת CI/CD - אין צורך בשילוב מיוחד, פשוט הוסיפו אותו ל-pipeline הקיים.

**תשובה מפורטת:**

BMAD לא דורשת שילוב מיוחד עם CI/CD - הפלט שלה הוא קבצי קוד רגילים, קונפיגורציות, וסקריפטים. אתם יכולים להוסיף אותם ל-Git ול-pipeline הקיים (Jenkins, GitHub Actions, GitLab CI, וכו') בדיוק כמו קוד שנכתב ידנית.

למעשה, BMAD עוזרת ליצור קונפיגורציות CI/CD טובות יותר: האג'נטים יודעים לכתוב pipeline scripts, לתכנן staging strategies, ולהגדיר automated testing. אם אתם בונים מערכת חדשה, אתם יכולים לבקש מ-Winston לתכנן לכם CI/CD pipeline מלא - הוא ייתן לכם קונפיג מוכן עם stages, tests, ו-deployment steps.

**דוגמה:**

סטארטאפ של ליאת מתחיל עם GitHub Actions. הם מריצים architecture workflow עם Winston ומבקשים ממנו להכין גם CI/CD pipeline. הוא כותב להם workflow.yml מלא עם: build, test, lint, security scan, ו-deploy לstagingץ הם מעתיקים את הקובץ ל-.github/workflows/ והכל עובד out of the box.

**קישורים רלוונטיים:**
